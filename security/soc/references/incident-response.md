## NIST SP 800-61（インシデントハンドリングガイド）

### Phase 1: Preparation（準備）
- IR ポリシー、計画、手順の策定
- IR チームの編成と訓練
- ツールの配備（フォレンジックワークステーション、通信チャネル、証拠保管）
- コミュニケーション計画の策定（内部・外部）
- テーブルトップ演習の実施

### Phase 2: Detection and Analysis（検出と分析）
- 攻撃ベクターの特定
- 前兆（Precursor）と指標（Indicator）の識別
- ソース: IDS/IPS、SIEM、AV、ファイル整合性チェック、ログ、人的報告
- インシデントの文書化と優先度付け
- 通知手順の実行

### Phase 3: Containment, Eradication, and Recovery（封じ込め・根絶・復旧）
- **封じ込め戦略**: 短期（即時隔離）と長期（恒久的修正準備中の一時対策）
- **証拠収集**: フォレンジック的に適切な手順で保全
- **根絶**: マルウェア除去、脆弱性パッチ適用
- **復旧**: バックアップからの復元、再感染の監視

### Phase 4: Post-Incident Activity（事後活動）
- 教訓会議（Lessons Learned）
- インシデントデータの活用
- 証拠の保持期間管理
- ポリシー・手順の更新

### インシデント重大度分類

| レベル | 機能的影響 | 情報的影響 | 復旧可能性 |
|--------|-----------|-----------|-----------|
| Critical | コア機能停止 | 機密データ大規模漏洩 | 復旧不可能または長期間 |
| High | 重要機能に大きな影響 | 機密データ漏洩 | 復旧に多大なリソース |
| Medium | 機能に中程度の影響 | 非公開データのアクセス | 予測可能な時間で復旧 |
| Low | 最小限の影響 | 影響なし | 即時復旧可能 |

## SANS PICERL

### 1. Preparation（準備）
- ポリシー、手順、コミュニケーション計画
- ツール・リソースの準備（ジャンプバッグ、フォレンジックWS）
- 訓練と演習
- 脅威インテリジェンスの統合

### 2. Identification（識別）
- インシデント発生の判断
- SIEM、IDS、EDR、ログからの証拠収集
- スコープと重大度の決定
- 初期所見の文書化

### 3. Containment（封じ込め）
- **短期封じ込め**: ネットワーク隔離、アカウント無効化
- **長期封じ込め**: 恒久対策準備中の一時修正
- **フォレンジックイメージング**: 根絶前の証拠保全

### 4. Eradication（根絶）
- マルウェア、バックドア、侵害アカウントの除去
- 根本原因脆弱性の特定とパッチ適用
- 必要に応じたシステム再構築
- スキャン・監視による根絶確認

### 5. Recovery（復旧）
- クリーンバックアップからの復元
- システム整合性の検証
- 強化された監視下での本番復帰
- 再侵害兆候の監視

### 6. Lessons Learned（教訓）
- 事後レビュー会議（2週間以内）
- タイムライン、対応、成功/失敗の文書化
- プレイブック・検出ルールの更新
- インテリジェンスの内部・コミュニティ共有

## Mandiant インシデントレスポンス手法

### Targeted Attack Lifecycle

```
初期偵察 → 初期侵害 → 足場確保 → 権限昇格 → 内部偵察 → 横展開 → 持続性維持 → 目的達成
```

### IR ライフサイクル

**Phase 1: Preparation**
- 事前準備性評価、テーブルトップ演習、Purple Team
- リテイナー契約による迅速展開体制

**Phase 2: Identification / Scoping**
- 初期トリアージとインテリジェンス収集
- 侵害範囲の特定（システム数、データ、アカウント）
- タイムライン構築

**Phase 3: Containment**
- C2 チャネルのブロック、侵害システムの隔離
- クレデンシャルリセット、ネットワークセグメンテーション
- **Quiet Containment**: 高度な攻撃者に対して検出を悟られない封じ込め

**Phase 4: Eradication**
- マルウェア、バックドア、持続性メカニズムの除去
- 初期アクセスベクターの封鎖
- 包括的スイープによる除去確認

**Phase 5: Recovery**
- 既知の正常バックアップからの復元
- 侵害インフラの再構築
- 段階的な本番接続
- 復旧期間中の強化監視

**Phase 6: Lessons Learned / Remediation**
- 詳細インシデントレポート
- 根本原因分析
- 戦略的改善ロードマップ

### 証拠収集方法論

**優先順位（揮発性順）:**
1. 揮発性データ（メモリ、実行中プロセス、ネットワーク接続）
2. ディスクフォレンジック（フルイメージまたはトリアージベース）
3. ログソース（EDR、Windows Event Log、Sysmon、FW/Proxy、DNS、Cloud Audit）
4. ネットワーク証拠（PCAP、NetFlow、DNS クエリログ）
5. クラウド証拠（Cloud Audit Trails、IAM 変更、ストレージアクセスログ）

**ツール:**
- Volatility（メモリフォレンジック）
- Plaso/log2timeline（スーパータイムライン作成）
- X-Ways（ディスクフォレンジック）
- YARA（マルウェア識別）

### フォレンジック分析アプローチ

1. **タイムライン分析**: 複数ログソースを統合したスーパータイムライン
2. **マルウェア分析**: 静的・動的分析、YARA ルール作成、サンドボックス
3. **インジケータ開発**: IOC 抽出（ハッシュ、IP、ドメイン、レジストリキー、Mutex）
4. **TTP マッピング**: MITRE ATT&CK への観察活動のマッピング
5. **帰属分析**: TTP、インフラ、マルウェアと既知脅威アクターの比較

### 改善戦略

| 時間軸 | レベル | 対応内容 |
|--------|--------|----------|
| 即時 | 戦術的 | IOC ブロック、侵害クレデンシャルリセット、システム隔離 |
| 短期 | 運用的 | 重大脆弱性パッチ、検出強化、露出サービス強化 |
| 長期 | 戦略的 | アーキテクチャ改善、ゼロトラスト、ログ/検出エンジニアリング改善 |

## インシデント対応プレイブック（共通パターン）

### フィッシング対応
1. メールヘッダー/URL/添付ファイル分析
2. サンドボックスでの動的分析
3. 類似メールの組織内検索
4. IOC 抽出とブロック
5. 影響ユーザーへの通知
6. メールボックスからの悪意あるメール削除

### ランサムウェア対応
1. **即時隔離**: 感染システムのネットワーク切断
2. スコープ評価（影響範囲の特定）
3. 暗号化状況の確認
4. バックアップ整合性の検証
5. 法執行機関への連絡
6. 身代金交渉の準備（必要な場合）
7. 復旧計画の策定と実行

### データ流出対応
1. DLP アラートのトリアージ
2. データ分類と影響評価
3. 流出スコープの特定
4. 規制通知要件の評価（個人情報保護法等）

### クラウドインシデント対応
1. IAM レビューとアクセスキーのローテーション
2. リソースインベントリの確認
3. 設定監査
4. クラウドネイティブログの分析
5. 侵害リソースの隔離・再構築
