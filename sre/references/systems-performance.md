# システムパフォーマンス リファレンス

Brendan Gregg「詳解 システム・パフォーマンス 第2版」に基づく SRE 実践ガイド

---

## 1. 基本概念

### 重要用語

| 用語 | 定義 |
|------|------|
| IOPS | 1秒あたりの I/O オペレーション数 |
| スループット | 仕事が実行されるスピード（データ転送速度 or オペレーション速度） |
| レイテンシ | オペレーションがサービス提供を待つ時間 |
| 応答時間 | オペレーション完了までの全時間（待ち+サービス+転送） |
| 使用率 | リソースがアクティブだった時間の割合 or 消費された容量 |
| 飽和度 | サービス提供できずキューイングされた作業の割合 |
| ボトルネック | システムパフォーマンスを制限しているリソース |

### タイムスケール（CPUサイクルを1秒とした対比）

| オペレーション | 実レイテンシ | 対比 |
|----------------|-------------|------|
| L1キャッシュ | 0.9ns | 3秒 |
| L2キャッシュ | 3ns | 10秒 |
| メインメモリ | 100ns | 6分 |
| SSD I/O | 10-100μs | 9-90時間 |
| 回転ディスクI/O | 1-10ms | 1-12ヶ月 |
| SF-NY間 | 40ms | 4年 |
| TCP再送タイムアウト | 1-3秒 | 105-317年 |

### Known-Unknowns フレームワーク

- **Known-Knowns**: 指標を知り値も知っている
- **Known-Unknowns**: チェックできるが未計測
- **Unknown-Unknowns**: 知らないことすら知らない

目標: Unknown-Unknowns を Known-Unknowns に変換する。

## 2. メソドロジ

### アンチパターン

| パターン | 症状 | 問題 |
|---------|------|------|
| 街灯 | 知っているツールだけを使う | 多くの問題を見過ごす |
| ランダム変更 | 手当たり次第にパラメータ変更 | 無意味なチューニングが残る |
| 他人のせい | データなしで他チームに投げる | 時間の浪費 |

### 問題の記述（最初に必ず行う6つの質問）

1. パフォーマンスに問題があると思ったのはなぜか?
2. 良好に動いていたことがあったか?
3. 最近の変化は何か?
4. レイテンシか実行時間で表現できるか?
5. ほかの人やアプリに影響しているか?
6. 環境はどうなっているか?

### 科学的メソッド

```
1. 問題を記述 → 2. 仮説 → 3. 検証方法設計 → 4. 検証 → 5. 分析 → 棄却なら2へ
```

**重要**: 1度に複数の変更を加えない。

### USEメソッド（最重要）

すべてのリソースの **使用率(U)、飽和度(S)、エラー(E)** をチェックする。

**手順:**
1. リソースのリストを作成
2. 各リソースについて E → S → U の順でチェック
3. 検出事項を精査
4. すべてのリソースをチェックしたか確認

**解釈ガイド:**
- 使用率100% → ボトルネック
- 使用率60%超 → 短期バーストが隠れている可能性
- 飽和度が0以外 → 問題の可能性
- エラーが0以外 → 精査すべき

### REDメソッド（マイクロサービス向け）

- **Rate**: リクエスト/秒
- **Errors**: 失敗リクエスト数
- **Duration**: リクエスト処理時間

USEはマシンの健全性、REDはユーザーから見た健全性。両者は補完関係。

### パフォーマンスマントラ（効果の高い順）

1. **するな** - 不要な仕事を取り除く
2. **二度するな** - キャッシング
3. **減らせ** - ポーリング頻度の削減
4. **先に延ばせ** - ライトバックキャッシング
5. **見られていないときにせよ** - ピーク外スケジューリング
6. **同時並行でせよ** - マルチスレッド化
7. **安上がりにせよ** - より高速なハードウェア

### レイテンシ分析（二分探索）

```
1. レイテンシを計測 → 2. A,Bに分割 → 3. 遅い方を計測 → 4. さらに分割 → 根本原因へ
```

## 3. USEメソッド チェックリスト（Linux）

### CPU

| 指標 | コマンド | 注意点 |
|------|---------|--------|
| 使用率 | `mpstat -P ALL 1` | CPU別に確認。1つだけ100%ならスレッドスケーリング問題 |
| 飽和度 | `vmstat 1`のr列、`cat /proc/pressure/cpu` | r > CPU数で飽和 |
| エラー | `dmesg \| grep -i "cpu\|throttl"` | スロットリング、ECC |

### メモリ

| 指標 | コマンド | 注意点 |
|------|---------|--------|
| 使用率 | `free -h`のavailable列 | フリーが少ない≠問題（キャッシュとして使用中は正常） |
| 飽和度 | `vmstat 1`のsi/so、`cat /proc/pressure/memory` | si/soが継続的に非ゼロ→問題 |
| エラー | `dmesg \| grep -i "out of memory"` | OOM Killer発動 |

### ディスク

| 指標 | コマンド | 注意点 |
|------|---------|--------|
| 使用率 | `iostat -sxz 1`の%util | 仮想ディスクではIOPS/スループットで判断 |
| 飽和度 | `iostat -sxz 1`のaqu-sz | キュー長が継続→飽和 |
| エラー | `smartctl -a /dev/sdX`、`dmesg` | S.M.A.R.T、カーネルエラー |

### ネットワーク

| 指標 | コマンド | 注意点 |
|------|---------|--------|
| 使用率 | `sar -n DEV 1` | 現在スループット/ネゴシエート速度 |
| 飽和度 | `ip -s link`のoverruns、TCP再送 | |
| エラー | `ip -s link`、`nstat` | ドロップ、CRC、フレームエラー |

## 4. 60秒パフォーマンス分析

障害対応の最初の60秒で実行すべきコマンド:

```bash
uptime                      # 1. ロードアベレージ（傾向把握）
dmesg -T | tail -20         # 2. カーネルエラー（OOM等）
vmstat -SM 1 5              # 3. CPU/メモリ/スワップ概要
mpstat -P ALL 1 3           # 4. CPU別偏りチェック
free -h                     # 5. メモリ使用状況
iostat -sxz 1 3             # 6. ディスクI/O（await最重要）
sar -n DEV 1 3              # 7. ネットワークスループット
sar -n TCP,ETCP 1 3         # 8. TCP再送
top -b -n 1 | head -20      # 9. プロセス別CPU/メモリ
cat /proc/pressure/{cpu,memory,io}  # 10. PSI（Linux 4.20+）
```

## 5. リソース別 詳細ガイド

### CPU

**ロードアベレージの解釈:**
- LA / CPU数 < 0.7: 余裕あり
- LA / CPU数 > 1.0: 飽和の可能性
- 1分/5分/15分の比較で傾向把握（増加中/減少中/安定）

**vmstat の読み方:**
- `r`: ランキュー長（r > CPU数 → 飽和）
- `us`/`sy`/`id`/`wa`/`st`: ユーザー/システム/アイドル/I/O待ち/スチール

**チューニング:**
- `nice`/`renice`: 優先度調整
- `taskset`: CPU バインド
- cgroup: CPU クォータ/シェア

### メモリ

**OOM Killer のメモリ開放順序:**
1. フリーリスト → 2. ページキャッシュ破棄 → 3. スワッピング → 4. リーピング → 5. OOM Kill

**重要パラメータ:**
| パラメータ | デフォルト | 説明 |
|-----------|----------|------|
| vm.swappiness | 60 | 0=キャッシュ優先破棄、高い値=アプリをスワップ |
| vm.overcommit_memory | 0 | 0=ヒューリスティック、2=オーバーコミット無効 |
| vm.dirty_ratio | 20 | ダーティメモリ%がこれを超えると書き戻し |

### ディスク

**iostat の最重要指標:**
- `await`: I/O要求の平均待ち時間（ms）→ レイテンシの直接指標
- `aqu-sz`: 平均キュー長 → 飽和度の指標
- `areq-sz`: 平均要求サイズ → < 8KB ならランダムI/O

### ネットワーク

**TCP再送:** パケット消失や輻輳の兆候。数千msのレイテンシを追加する。

**Netflix本番環境のsysctlチューニング:**
```bash
net.core.somaxconn = 1024
net.core.netdev_max_backlog = 5000
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_congestion_control = bbr
net.ipv4.tcp_max_syn_backlog = 8192
net.ipv4.tcp_rmem = 4096 12582912 16777216
net.ipv4.tcp_wmem = 4096 12582912 16777216
net.ipv4.tcp_slow_start_after_idle = 0
net.ipv4.tcp_tw_reuse = 1
net.ipv4.ip_local_port_range = 10240 65535
```

## 6. 統計の基礎

### 平均の限界
- ディスクI/Oが平均1msでも外れ値は100ms超
- 5分平均では数秒間のCPU 100%が見えない
- キャッシュヒット/ミスで二峰分布になることが多い

### 使うべき統計量
- **パーセンタイル**: p50, p90, p95, p99, p99.9
- **ヒストグラム**: 分布全体を把握
- **ヒートマップ**: レイテンシ分布の時系列変化

### 待ち行列理論
- 使用率 60% 超で平均応答時間が2倍
- 使用率 80% 超で3倍
- 100%前に問題が顕在化する

## 7. 可観測性ツール

### ツール分類

| | カウンタ（低コスト） | イベントベース（高コスト） |
|---|---|---|
| システム全体 | vmstat, mpstat, iostat, sar | perf, Ftrace, BCC, bpftrace |
| プロセス | ps, top, pidstat | strace, perf(-p PID) |

### クライシスツール（事前インストール必須）

```
procps, util-linux, sysstat, iproute2, linux-tools, bcc-tools, bpftrace
```

### CPUプロファイリング

```bash
# perf でプロファイル取得 → フレームグラフ
perf record -F 49 -a -g -- sleep 30
perf script | stackcollapse-perf.pl | flamegraph.pl > cpu.svg
```

**フレームグラフの読み方:**
- フレームの幅 = CPU時間に比例
- 大きな「高原」と「塔」に注目
- `ext4`/`blk` → ファイル/ブロックI/O、`TCP` → ネットワーク、`utex` → ロック競合

### off-CPU分析

```bash
offcputime -f -p PID 30 > offcpu.stacks
flamegraph.pl --color=io < offcpu.stacks > offcpu.svg
```

## 8. BPF/BCC/bpftrace

### BCC 必携ツール

| ツール | 機能 |
|--------|------|
| execsnoop | 新プロセス作成トレース |
| biolatency | ディスクI/Oレイテンシ分布 |
| biosnoop | ディスクI/Oイベントトレース |
| runqlat | CPUランキューレイテンシ |
| profile | CPUスタックプロファイリング |
| offcputime | off-CPU時間のスタック別集計 |
| tcplife | TCP接続ライフサイクル |
| tcpretrans | TCP再送トレース |
| cachestat | ファイルキャッシュヒット率 |
| ext4slower | 遅いFS操作トレース |
| memleak | メモリリーク検出 |

### bpftrace ワンライナー

```bash
# 新プロセスをトレース
bpftrace -e 'tracepoint:syscalls:sys_enter_execve { join(args->argv); }'

# システムコール別カウント
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[pid, comm] = count(); }'

# カーネルスタック99Hzサンプリング
bpftrace -e 'profile:hz:99 /pid/ { @[kstack] = count(); }'

# ブロックI/Oサイズ分布
bpftrace -e 't:block:block_rq_issue { @bytes = hist(args->bytes); }'

# TCP送信バイト数分布
bpftrace -e 'k:tcp_sendmsg { @send_bytes = hist(arg2); }'

# ファイルオープントレース
bpftrace -e 't:syscalls:sys_enter_openat { printf("%s %s\n", comm, str(args->filename)); }'

# VFS readレイテンシ
bpftrace -e 'k:vfs_read { @ts[tid] = nsecs; } kr:vfs_read /@ts[tid]/ {
    @ = hist(nsecs - @ts[tid]); delete(@ts[tid]); }'
```

## 9. コンテナのパフォーマンス

### コンテナ内ツールの落とし穴

| ツール | コンテナ内での挙動 |
|--------|-------------------|
| top, uptime, mpstat, vmstat, free, iostat | **ホスト全体**の情報を表示 |
| ps, pidstat | コンテナのプロセスのみ（正しい） |
| perf | 実行失敗 or 他テナントもプロファイリング |

### CPU抑制の確認

```bash
cat /sys/fs/cgroup/cpu/cpu.stat
# nr_throttled: 抑制回数
# throttled_time: 抑制時間合計（ns）
```

### cgroups リソース制御

| cgroup | 機能 |
|--------|------|
| cpu | シェア（下限保証）、帯域幅（上限制限） |
| cpuset | 特定CPU割り当て |
| memory | メモリ上限、OOM制御 |
| blkio | I/O制限 |

## 10. ベンチマーキング

### まずいベンチマークの16パターン（抜粋）

1. テスト対象の誤り（実ワークロードと無関係）
2. 環境の無視（テスト≠本番）
3. エラーの無視（全リクエストエラーなのに結果を信じる）
4. ばらつきの無視（平均のみ、バースト未考慮）
5. 複数要素の同時変更

### アクティブベンチマーキング

```
1. 実行中にUSEメソッドで制約要因を特定
2. CPUプロファイリングでベンチマーク自身を分析
3. ワークロード特性を確認（本当にテスト対象をテストしているか）
4. 制約要因と分析結果を結果に付記
```

## 11. アラート閾値ガイドライン

| メトリクス | 注意 | 警告 | 危険 |
|-----------|------|------|------|
| CPU使用率（平均） | > 70% | > 85% | > 95% |
| ロードアベレージ/CPU数 | > 0.7 | > 1.0 | > 2.0 |
| メモリ available | < 20% | < 10% | < 5% |
| スワップ | > 0 | 持続的 si/so | OOM |
| ディスク %util | > 60% | > 80% | 100% 2秒以上 |
| ディスク await | 基準の2倍 | 基準の5倍 | > 100ms |
| ネットワーク使用率 | > 50% | > 70% | > 90% |
| TCP再送率 | > 0.1% | > 1% | > 5% |

## 12. 障害対応フロー

```
障害発生
  ↓
[問題の記述] 6つの質問に答える
  ↓
[60秒分析] uptime → dmesg → vmstat → mpstat → iostat → free → sar → top
  ↓
[USEメソッド] 全リソースのU/S/Eをチェック
  ├→ ボトルネック発見 → [ドリルダウン分析]
  └→ 見つからない → [REDメソッド] → [ワークロード特性把握] → [CPUプロファイリング]
  ↓
[レイテンシ分析] 二分探索で根本原因に到達
  ↓
[修正と定量化] レイテンシ削減量で効果を測定
```
