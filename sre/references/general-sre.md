# SRE ベストプラクティス（汎用）

## 1. オブザーバビリティ

### メトリクスの基本型

| 型 | 説明 | 用途例 |
|---|---|---|
| Counter | 単調増加する累積値 | リクエスト数、エラー数 |
| Gauge | 任意に増減する瞬時値 | CPU使用率、キュー長 |
| Histogram | 値の分布をバケットに集計 | レスポンスタイム |
| Summary | クライアント側でパーセンタイル計算 | レイテンシ（集約不可） |

### RED メソッド（サービス向け）
- **Rate**: 1秒あたりのリクエスト数
- **Errors**: 失敗したリクエストの割合
- **Duration**: リクエスト処理時間の分布

### USE メソッド（リソース向け）
- **Utilization**: リソースがビジーな時間割合
- **Saturation**: 処理しきれない超過作業量
- **Errors**: エラーイベント数

### ログレベル

| レベル | 用途 | 本番環境 |
|---|---|---|
| FATAL | プロセス継続不能 | 常に出力 |
| ERROR | 処理失敗、対応が必要な可能性 | 常に出力 |
| WARN | 異常だが処理継続可能 | 常に出力 |
| INFO | 正常な重要イベント | 常に出力 |
| DEBUG | トラブルシューティング詳細 | 通常無効、動的に有効化 |

### 分散トレーシング

```

├─ [Span: API Gateway] 0ms ──────────── 150ms
│  ├─ [Span: Auth Service] 5ms ── 25ms

│     ├─ [Span: DB Query] 35ms ── 50ms

```

- ベンダーロックインを避け、バックエンドを交換可能に

## 2. アラート設計

### 分類
```
Page（即時対応）: ユーザー影響がある障害、SLO バーンレート高、データ損失危険
Ticket（営業時間内）: パフォーマンス低下兆候、リソース枯渇予兆
Log/Dashboard（情報記録）: 自動復旧イベント、キャパシティトレンド
```

### SLO ベースのバーンレートアラート

```promql
# 月間SLO 99.9%、1時間で2%のバジェット消費（14.4x）→ Page
(
  job:sli_errors:ratio_rate1h{job="api-server"} > (14.4 * 0.001)
  and
  job:sli_errors:ratio_rate5m{job="api-server"} > (14.4 * 0.001)
)
```

## 3. インシデント管理

### 重大度レベル

| レベル | 影響 | 対応時間 | 例 |
|---|---|---|---|
| SEV1 | 全ユーザー | 即座（15分以内） | 全サーバーダウン、セキュリティ侵害 |
| SEV2 | 多数ユーザー | 30分以内 | 決済停止、レイテンシ10倍 |
| SEV3 | 一部ユーザー | 4時間以内 | 特定画面でエラー |
| SEV4 | 最小限 | 次営業日 | UI表示崩れ |

### インシデント役割
- **Incident Commander (IC)**: 方針決定、リソース割当、エスカレーション判断
- **Operations Lead**: 技術的調査と復旧作業
- **Communications Lead**: ステークホルダー報告、ステータスページ更新
- **Scribe**: タイムライン記録

### インシデントライフサイクル
```
検出 → トリアージ → 調査 → 緩和 → 解決 → クローズ → ポストモーテム
```

**原則:**
- 「まず止血（緩和）、次に原因調査」
- ロールバックが最も早い復旧手段であることが多い
- 15-30分ごとにステータスを更新

### ポストモーテムテンプレート

```markdown
# ポストモーテム: [タイトル]
## 概要（日時、重大度、影響、IC、関係チーム）
## タイムライン（時刻 | イベント）
## 根本原因
## 検出（方法、TTD）
## 緩和と解決（TTM、TTR）
## うまくいったこと / 改善すべきこと
## アクションアイテム（# | アクション | 担当 | 期限 | 優先度）
## 教訓
```

## 4. パフォーマンスエンジニアリング

### なぜ平均値が危険か

```
100リクエスト: 95件→50ms, 4件→200ms, 1件→5000ms
平均: 120ms（一見「問題なし」）
p99: 5000ms（100人に1人が体験する深刻な遅延）
```

### パーセンタイルの使い分け
| パーセンタイル | 用途 |
|---|---|
| p50 | 典型的なユーザー体験 |
| p95 | SLO の一般的な閾値 |
| p99 | テールレイテンシ、インフラ問題検出 |

### 負荷テスト

| 種類 | 目的 |
|------|------|
| ベースラインテスト | 通常負荷での性能把握 |
| 負荷テスト | ピーク時の性能確認 |
| ストレステスト | 限界性能の把握 |
| スパイクテスト | 急激な負荷変動への耐性 |
| ソークテスト | 長時間運用の問題検出（メモリリーク等） |

### キャパシティプランニング
- 需要予測: 季節性、成長率、特別イベント
- リソースモデリング: 需要とリソース消費の定量化
- ヘッドルーム: 通常時50-60%、ピーク時でも80%以下
- 四半期ごとにプランを見直し

## 5. データベース信頼性

### コネクションプーリング
- プールサイズ目安: `(コア数 * 2) + ディスクスピンドル数`
- プロキシ型プーラー（PgBouncer、ProxySQL）で接続多重化
- コネクションリークを防止（defer close）

### リードレプリカ
- Write → プライマリ、Read → レプリカ
- レプリケーションラグを常時モニタリング
- 書き込み直後の読み取りはプライマリから（Read-your-writes）

### バックアップ
```
3-2-1 ルール:
- 3つのコピーを保持
- 2つの異なるメディアに保存
- 1つはオフサイトに保存
```

- 月1回以上のリストアテスト実施
- RPO を明確に定義

### 安全なスキーママイグレーション
- カラム追加: nullable またはデフォルト値付き
- カラム削除: 3段階（参照削除 → nullable化 → 物理削除）
- インデックス追加: `CREATE INDEX CONCURRENTLY`
- 大規模データ変更: バッチ処理で少しずつ

## 6. ランブック

### 構造

```markdown
# ランブック: [アラート名]
## 概要（対象アラート、サービス、重大度、オーナー）
## 影響
## 前提条件（アクセス権限等）
## 診断手順（Step 1, 2, 3...）
## 対応手順（シナリオA, B, C...）
## エスカレーション
## 関連リンク
```

### トラブルシューティング決定木

```
APIレイテンシ高騰
├── 最近デプロイがあったか？ → YES → ロールバック
├── 特定エンドポイントのみ？ → YES → 依存先を調査
├── リソース逼迫？ → YES → スケールアウト
├── ネットワーク問題？ → DNS/TCP/パケットロス調査
└── 上記すべて非該当 → プロファイリング + エスカレーション
```

### 自動化の段階
1. 手動ランブック
2. 支援付き自動化（診断情報自動収集）
3. 半自動化（定型対応を自動化、人間が承認）
4. 完全自動化（検出から復旧まで自動）

### 一般的な障害パターン

**メモリリーク:**
- 検出: メモリ単調増加、OOMKill
- 短期: Pod再起動 → 中期: プロファイリング → 長期: コード修正

**カスケード障害:**
- 検出: 1サービスの障害が連鎖波及
- 対応: サーキットブレーカー有効化、障害サービスへのトラフィック遮断

**ディスク枯渇:**
- 検出: ディスク使用率アラート
- 対応: 不要ファイル削除 → ログローテーション確認 → ディスク拡張
- 予防: 段階的アラート（70%, 80%, 90%）

**証明書期限切れ:**
- 予防: 30日/14日/7日前にアラート
- cert-manager / Let's Encrypt による自動更新
