# Google SRE プラクティス

## 1. SRE の基本原則

### 定義
SRE（Site Reliability Engineering）は、Google の Ben Treynor Sloss が2003年に提唱した概念。

> 「SRE とは、ソフトウェアエンジニアに運用の設計を依頼したときに出来上がるもの」

**核心的な考え方:**
- 運用作業をソフトウェアエンジニアリングで解決する
- 手作業（Toil）を自動化で置き換える
- 信頼性を「機能」として扱う
- データ駆動の意思決定を行う

### SRE と DevOps の関係

> 「class SRE implements DevOps」

| 観点 | DevOps | SRE |
|------|--------|-----|
| 位置づけ | 文化・哲学 | 具体的な実践・フレームワーク |
| フォーカス | Dev と Ops の壁を壊す | 信頼性をエンジニアリングで担保する |
| 測定 | DORA メトリクス等 | SLI/SLO/エラーバジェット |

## 2. SLI / SLO / SLA

### SLI（Service Level Indicator）

サービスの振る舞いを定量的に測定する指標。

**計算式:**
```
SLI = (良好なイベントの数 / 有効なイベントの総数) × 100%
```

**サービスタイプ別の推奨 SLI:**

| サービスタイプ | SLI の種類 | 測定方法の例 |
|--------------|-----------|------------|
| ユーザー向け Web | 可用性 | HTTP 200 レスポンスの割合 |
| ユーザー向け Web | レイテンシ | 200ms 以内に応答したリクエストの割合 |
| ストレージ | 耐久性 | 正常に読み取れたデータの割合 |
| データパイプライン | 鮮度 | 閾値以内の遅延で処理されたデータの割合 |
| データパイプライン | 正確性 | 正しく処理されたレコードの割合 |

### SLO（Service Level Objective）

SLI に対する目標値。

**設定のベストプラクティス:**
1. ユーザーの期待に基づいて設定する
2. 達成可能だが挑戦的な値にする
3. 段階的に厳しくする
4. すべてのステークホルダーの合意を得る

**具体例:**
```
- 可用性 SLO: 30日間のローリングウィンドウで 99.9% のリクエストが成功する
- レイテンシ SLO: 30日間のローリングウィンドウで 99% のリクエストが 200ms 以内に応答する
```

### SLA（Service Level Agreement）

SLO を満たせなかった場合のペナルティを含むビジネス契約。SLA は SLO より緩く設定する。

### SLI / SLO / SLA の関係
```
SLI: 「現在のレイテンシの p99 は 150ms です」（測定値）
SLO: 「レイテンシの p99 を 200ms 以下に保ちます」（内部目標）
SLA: 「レイテンシの p99 が 300ms を超えた場合、月額料金の 10% を返金します」（契約）
```

## 3. エラーバジェット

### 概念

> **100% の可用性は間違った目標である。** ユーザーは 99.999% と 100% の違いを認識できない。

**計算式:**
```
エラーバジェット = 1 - SLO
```

**許容ダウンタイム（30日間）:**

| SLO | エラーバジェット | 許容ダウンタイム |
|-----|----------------|----------------|
| 99.0% | 1.0% | 7時間12分 |
| 99.5% | 0.5% | 3時間36分 |
| 99.9% | 0.1% | 43分12秒 |
| 99.95% | 0.05% | 21分36秒 |
| 99.99% | 0.01% | 4分19秒 |
| 99.999% | 0.001% | 26秒 |

### エラーバジェットポリシーの例

```
1. 残量 50% 以下: リリース頻度を半分に、追加レビューを要求
2. 残量 25% 以下: 信頼性に直接寄与しない変更のリリースを停止
3. 使い切った場合: すべてのリリースを凍結（セキュリティ修正を除く）
```

## 4. ゴールデンシグナル

すべてのシステムで監視すべき4つの重要な指標。

### 1. レイテンシ（Latency）
- 成功とエラーのレイテンシを分けて計測する
- パーセンタイル（p50, p90, p95, p99）を監視する

### 2. トラフィック（Traffic）
- Web: RPS、ストリーミング: I/O レート、ストレージ: トランザクション/秒

### 3. エラー（Errors）
- 明示的エラー（HTTP 5xx）、暗黙的エラー（200だが内容不正）、ポリシーベースエラー

### 4. サチュレーション（Saturation）
- CPU、メモリ、ディスク I/O、ネットワーク帯域の使用率
- 多くのシステムは 70-80% 超過で劣化が始まる

---

## 5. 監視手法

### RED メソッド（サービス向け）
- **R**ate（レート）: リクエスト/秒
- **E**rrors（エラー）: エラー/秒
- **D**uration（所要時間）: レイテンシの分布

### USE メソッド（リソース向け）
- **U**tilization（利用率）
- **S**aturation（飽和度）
- **E**rrors（エラー）

## 6. アラート設計

### 症状ベース vs 原因ベース

| アプローチ | 説明 | 例 |
|-----------|------|-----|
| 症状ベース（推奨） | ユーザー影響を検知 | 「エラーレートが 1% を超えた」 |
| 原因ベース（非推奨） | 特定の原因を検知 | 「CPU 使用率が 90% を超えた」 |

### アラートの3つの出力先

| 出力先 | 条件 |
|--------|------|
| ページ（即時対応） | 今すぐ人間が行動する必要がある |
| チケット | 数日以内に対応が必要 |
| ログ/ダッシュボード | 参考情報として記録 |

### バーンレートアラート

```
バーンレート = 実際のエラー率 / SLO で許容されるエラー率
```

**推奨設定（月間SLO 99.9%）:**

| 重大度 | ロングウィンドウ | ショートウィンドウ | バーンレート |
|--------|---------------|-----------------|------------|
| P1（ページ） | 1時間 | 5分 | 14.4x |
| P2（ページ） | 6時間 | 30分 | 6x |
| P3（チケット） | 3日 | 6時間 | 1x |

## 7. インシデント管理

### インシデントのライフサイクル
```
検知 → トリアージ → 対応 → 復旧 → 事後対応（ポストモーテム）
```

### 重大度レベル

| レベル | 定義 | 例 |
|--------|------|-----|
| P0 | サービスの完全停止、データ損失の可能性 | 全ユーザーがログインできない |
| P1 | 主要機能の大幅な劣化 | 決済処理が50%失敗 |
| P2 | 一部機能への影響 | 検索結果の表示が遅い |
| P3 | 軽微な影響 | 管理画面の一部グラフが表示されない |

### インシデント対応の役割

| 役割 | 責任 |
|------|------|
| インシデントコマンダー（IC） | 全体統括、意思決定 |
| コミュニケーションリード | ステークホルダーへの状況共有 |
| オペレーションリード | 技術的な調査・対応の実行 |

### コミュニケーションテンプレート
```
【インシデント更新 - {時刻}】
■ 現在の状況: {概要}
■ 影響範囲: {影響を受けているユーザー/サービス}
■ 現在の対応: {実施中のアクション}
■ 次のステップ: {計画しているアクション}
■ 次回更新: {時刻}
```

## 8. ブレイムレスポストモーテム

### 基本原則
> 「人を責めるのではなく、システムを改善する」

### 実施すべき条件
- ユーザーに影響があったインシデント
- エラーバジェットを大きく消費したインシデント
- 解決に一定時間以上かかったインシデント
- 監視によって検知されなかったインシデント

### ポストモーテム文書の構造

```markdown
# ポストモーテム: {タイトル}

## メタデータ
- 日付 / 作成者 / 重大度 / インシデント期間

## サマリー（2-3文）
## 影響（ユーザー数、サービス、エラーバジェット消費）
## 根本原因
## トリガー
## 検知（方法、TTD）

## タイムライン
| 時刻 | イベント |

## 対応と復旧（TTM、TTR）
## うまくいったこと
## うまくいかなかったこと
## 幸運だったこと

## アクションアイテム
| アクション | 種類（prevent/detect/mitigate） | 担当者 | 期限 |

## 教訓
```

## 9. トイル（Toil）

### 定義
以下の特性をすべて持つ作業:
- 手動的 / 繰り返し的 / 自動化可能 / 戦術的 / 価値が持続しない / O(n)で増加

### 50% ルール
SRE はトイルに費やす時間を50%以下に抑え、残り50%以上をエンジニアリングプロジェクトに使う。

### トイル削減戦略
1. 完全自動化
2. セルフサービス化
3. 排除（不要にする設計変更）
4. ポリシー変更
5. 頻度の削減

---

## 10. リリースエンジニアリング

### 原則
1. セルフサービスモデル
2. 高い再現性（Hermetic Builds）
3. 自動化
4. 段階的ロールアウト

### カナリアリリースの段階
```
段階1: 1% → 15分監視
段階2: 10% → 30分監視
段階3: 50% → 1時間監視
段階4: 100% → 継続監視
```

---

## 11. キャパシティプランニング

### 必要キャパシティの計算
```
必要キャパシティ = ピークトラフィック × 安全係数 / (1 - 障害時の縮退分)

例: ピーク 10,000 RPS × 1.5 / 1,000 RPS/台 + 2台(N+2) = 17台
```

## 12. オンコール

### 推奨構成
- 最低8人のローテーション
- プライマリ + セカンダリの2層構造
- 1シフトあたりのインシデント対応: 最大2件
- オンコール比率: 最大25%

### バーンアウト防止
- ノイジーなアラートの削減
- ランブックの整備
- 深夜対応後の代休
- オンコール負荷のトラッキング

---

## 13. SRE 成熟度モデル

| レベル | 段階 | 特徴 |
|--------|------|------|
| 0 | 反応的 | 場当たり的対応、SLO なし |
| 1 | 基礎的 | SLI/SLO 定義済、基本的監視、ポストモーテム実施 |
| 2 | 体系的 | エラーバジェット運用、トイル計測・削減、オンコール整備 |
| 3 | 予防的 | カオスエンジニアリング、キャパシティ自動化、SLO ベースアラート |
| 4 | 最適化 | SLO チューニング、自動修復、組織全体に SRE 文化浸透 |
