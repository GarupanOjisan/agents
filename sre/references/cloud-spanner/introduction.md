# はじめに

Cloud Spanner の最も基本的な知識を共有する。

## 基礎編

### Cloud Spanner とは

Google Cloud が提供する高可用性な分散 RDB。

### 従来の RDB との違い

#### 1. シャーディング不要で無限にスケールする

従来の RDB で行っていたようなアプリケーション層でのシャーディングや、Master - Read Replica へのクエリ振り分けが不要。代わりに Cloud Spanner は **Processing Unit（PU: 計算ユニット）** を変更するだけで処理能力をスケールできる。

#### 2. 高可用性

従来の RDB だと複数のインスタンスにデータを複製して高可用性を担保し、インフラエンジニアが継続的に監視する必要があった。Cloud Spanner では Google Cloud のデータセンター間で自動的にデータが複製され高可用性が担保されるため、開発者は従来のような努力をせずにサービスを安定運用できる。

## 応用編

### PU: Processing Unit

Cloud Spanner の処理能力は PU またはノード数（= 1,000 PU）で表現される。

- **1,000 PU 未満**: 開発環境などコスト重視の構成向け
- **1,000 PU 以上**: SLA 対象。本番環境では基本的にこちら

### データストレージの上限

PU（またはノード）ごとに扱えるデータストレージのサイズに上限がある。データ量が増えると必然的に PU を上げる必要があるため、コスト増につながる。Spanner に保存しなくて良いデータは他の場所に保存するとよい。

PU は勝手には上がらないので、サードパーティの [Spanner Autoscaler](https://github.com/cloudspannerecosystem/autoscaler) 相当を導入するか、Google マネージドのオートスケーラーを使う（[オートスケーリング](./autoscaling.md) を参照）。
