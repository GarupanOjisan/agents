## 1. スキーマ設計 & データモデリング

### 1.1 プライマリキー設計

**鉄則: 単調増加キーを絶対に避ける。**

Spanner はデータをキーレンジ単位の「スプリット」に分割し、各スプリットを異なるノードに割り当てる。タイムスタンプや連番を先頭キーにすると、すべての新規書き込みがキー空間の末尾に集中する（**Moving Hotspot** 問題）。

| アプローチ | 構文例（GoogleSQL） | 特徴 |
|------------|---------------------|------|
| UUID v4 | `FanId STRING(36) NOT NULL DEFAULT (GENERATE_UUID())` | 最もシンプル。衝突確率は 2^122 分の1。Spanner に UUID 型は存在しないため STRING(36) で格納 |
| ビット反転シーケンス（IDENTITY） | `GENERATED BY DEFAULT AS IDENTITY (BIT_REVERSED_POSITIVE_SEQUENCE)` | 整数IDが必要な場合。既存DBからの移行に適する |
| ビット反転シーケンス（明示 SEQUENCE） | `CREATE SEQUENCE` → `DEFAULT (GET_NEXT_SEQUENCE_VALUE(...))` | シーケンス定義を独立管理したい場合 |
| ハッシュプレフィックス | `FARM_FINGERPRINT(ActualKey) % N AS ShardId` | Nをノード数に合わせて設定 |

**定量的閾値**: 1スプリットが処理できる書き込みは約 **3,500 writes/秒**。超える見込みがあれば必ず分散キーを選ぶ。

**落とし穴**:
- **UUIDv7 は危険**: タイムスタンププレフィックスを持つため、単調増加キーと同等の問題を引き起こす
- **シャード数 N は静的**: `FARM_FINGERPRINT % N` のNを後から変えるにはデータ再書き込みが必要

```sql
-- BAD: 連番主キー
CREATE TABLE Bad (Id INT64 NOT NULL) PRIMARY KEY (Id);

-- GOOD: UUID v4（STRING(36) に格納、GENERATE_UUID() で自動生成）
CREATE TABLE Good (
  Id STRING(36) NOT NULL DEFAULT (GENERATE_UUID())
) PRIMARY KEY (Id);

-- GOOD: ビット反転シーケンス（IDENTITY 句）
CREATE TABLE Better (
  Id INT64 NOT NULL GENERATED BY DEFAULT AS IDENTITY (BIT_REVERSED_POSITIVE_SEQUENCE)
) PRIMARY KEY (Id);

-- GOOD: ビット反転シーケンス（明示 SEQUENCE）
CREATE SEQUENCE BetterIdSeq OPTIONS (sequence_kind = 'bit_reversed_positive');
CREATE TABLE BetterExplicit (
  Id INT64 NOT NULL DEFAULT (GET_NEXT_SEQUENCE_VALUE(SEQUENCE BetterIdSeq))
) PRIMARY KEY (Id);

-- GOOD: ハッシュプレフィックス + タイムスタンプ
CREATE TABLE TimeSeries (
  ShardId INT64 NOT NULL,
  Timestamp TIMESTAMP NOT NULL,
  UserId STRING(36) NOT NULL,
) PRIMARY KEY (ShardId, Timestamp DESC);
```

### 1.2 インターリーブテーブル

親子関係のデータを同一スプリットに物理的に同居させ、分散JOINを排除する。

```sql
CREATE TABLE Singers (
  SingerId INT64 NOT NULL,
  Name STRING(1024),
) PRIMARY KEY (SingerId);

CREATE TABLE Albums (
  SingerId INT64 NOT NULL,
  AlbumId INT64 NOT NULL,
  Title STRING(1024),
) PRIMARY KEY (SingerId, AlbumId),
  INTERLEAVE IN PARENT Singers ON DELETE CASCADE;
```

**使うべき場面**:
- 親と子を一緒に読む操作が全体の80%以上
- 書き込み時に親子を同一トランザクションでまとめて書く
- 子テーブルの主要アクセスが「特定の親に属する子一覧」

**避けるべき場面**:
- 全親にわたって子テーブルを水平スキャン（逆に遅くなる）
- 親1件に対して子行が数万〜数百万行（親+子を 1 つの Split に収めきれないほど肥大化。Split は一般的に数 GB 規模で動的分割されるが、Google は厳密な上限値を公開していない）
- 多対多関係（インターリーブは1親のみ）

**定量的効果**:
- 親子JOIN P99レイテンシ: **3ms** vs なし **15ms**（5倍改善）
- 書き込みレイテンシ: 7ms → 3ms（2PC参加者 2→1）

**制約**:
- 最大深度 **7レベル**（根テーブルを含む）
- TTLを使う親テーブルのインターリーブ子は **ON DELETE CASCADE** が必須

### 1.3 スキーマ変更（DDL）

スキーマ変更はダウンタイムゼロで実行可能だが、種類により所要時間が異なる。

| 操作 | 所要時間 | バックフィル |
|------|---------|-------------|
| 列追加（NULL許可） | 数分 | なし |
| テーブル + インデックス同時作成 | 数分 | なし |
| 既存テーブルへのインデックス追加 | 数分〜数時間 | あり |
| NOT NULL 制約追加 | 数分〜数時間 | あり（全行スキャン） |
| 外部キー追加（enforced） | 長時間 | あり |

**運用ルール**:
1. バックフィルが必要なインデックスは **1日3本まで** が推奨
2. テーブルとインデックスは **同一バッチで同時に作成**（別々だと2スキーマバージョンが生成）
3. **インデックス作成前にノードを増設**（開始後の増設は並列度に反映されない）
4. **バルクロード後にインデックスを作成**（ロード中のインデックス更新を回避）
5. スキーマオブジェクト上限: テーブル 5,000、インデックス 10,000、カラム 1,024/テーブル、PK列 16

**NOT NULL 追加の落とし穴**: バリデーション完了前でもNULL書き込みがほぼ即座に拒否される。

### 1.4 カラム型の選択

**NUMERIC vs FLOAT64**: 金融・会計データには必ず NUMERIC を使う。

| | NUMERIC | FLOAT64 |
|--|---------|---------|
| 0.2の格納 | 正確に 0.2 | 0.20000000000000001 |
| 精度 | 38桁、小数点以下9桁 | 64ビット浮動小数点（近似値） |

**JSON カラム**:
- インデックス不可 → 生成カラムで値を抽出してインデックス化
- ネスト深度上限: 80レベル
- コアフィールドはスキーマカラム化、可変データのみJSONに（ハイブリッドアプローチ）

**ARRAY カラム**:
- `ARRAY<ARRAY<T>>` は不可 → `ARRAY<STRUCT<field ARRAY<T>>>` で代替
- 子行が多い場合は別テーブル+インターリーブの方がスケーラブル

### 1.5 セカンダリインデックス

**STORING 句（カバリングインデックス）**: ベーステーブルへのバックジョインを回避。

```sql
CREATE INDEX AlbumsByTitle ON Albums(Title) STORING (ReleaseDate, MarketingBudget);
```

**NULL_FILTERED インデックス**: スパースカラムでサイズ大幅削減、書き込みオーバーヘッド低減。

```sql
CREATE NULL_FILTERED INDEX AlbumsByTitle ON Albums(Title);
```

**コミットタイムスタンプをインデックス先頭に置かない**:

```sql
-- 危険: ホットスポット発生
CREATE INDEX CommitIdx ON Events(CommitTimestamp);
-- 安全: 先頭に分散キー
CREATE INDEX CommitIdx ON Events(EventType, CommitTimestamp);
```

### 1.6 外部キー vs インターリーブ

| 観点 | インターリーブ | FK（Enforced） | FK（Informational） |
|------|------------|---------------------|--------------------------|
| 物理的データ共存 | あり | なし | なし |
| 参照整合性の強制 | 暗黙 | あり | なし |
| 書き込みオーバーヘッド | 低 | 高 | 低 |
| 複数親テーブル参照 | 不可 | 可 | 可 |

### 1.7 生成カラム（Generated Columns）

```sql
-- ケースインセンシティブ検索
ALTER TABLE Products ADD COLUMN UpperName STRING(MAX) AS (UPPER(Name)) STORED;
CREATE INDEX ProductsByUpperName ON Products(UpperName);

-- JSON フィールドのインデックス化
ALTER TABLE Events ADD COLUMN EventType STRING(100)
  AS (JSON_VALUE(Payload, '$.type')) STORED;
CREATE INDEX EventsByType ON Events(EventType);
```

**制約**: 同一テーブル内の列のみ参照可能、非決定論的関数は STORED 不可、式の変更不可（削除→再作成が必要）。

### 1.8 TTL（行削除ポリシー）

```sql
ALTER TABLE UserEvents
ADD ROW DELETION POLICY (OLDER_THAN(EventTime, INTERVAL 30 DAY));
```

**削除タイムライン**:
| 段階 | タイムライン |
|------|------------|
| TTL期限到来 | クエリには引き続き見える |
| 実際の削除実行 | **最大72時間以内** |
| ストレージ解放 | 削除から**最大7日以内** |

**トゥームストーン問題（最重要の落とし穴）**: 削除された行はバージョン保持期間中トゥームストーンとして残りスキャン対象になる。

実測例:
- 削除前の `SELECT MIN(ts)` → **16.56ms**
- 2,000万行削除後 → **7.09秒**

対策: クエリに `WHERE ts > [削除閾値]` を追加して削除済み範囲を除外。初回大量削除はPartitioned DMLで先行実施。

**その他の注意点**:
- テーブルあたり **1つのみ**
- 間隔は **整数日単位のみ**
- バックアップ復元後は **TTLポリシーが自動削除** → 手動再設定が必要
- Change Streamsで監査可能（`transaction_tag = 'RowDeletionPolicy'`）

## 2. クエリパフォーマンス & 最適化

### 2.1 ステール読み取り（Stale Reads）

| 種類 | 説明 | レイテンシ |
|------|------|----------|
| Strong Read | 常に最新データ | リーダーへのラウンドトリップ必須 |
| Bounded Staleness | 指定古さの上限内 | ネゴシエーション発生 |
| Exact Staleness | 指定タイムスタンプ | ネゴシエーション不要、最速 |

**推奨**: **15秒以上のExact Staleness**。リーダーは約10秒ごとにレプリカの安全タイムスタンプを更新するため、15秒ステールで近傍レプリカから直接読み取り可能。

**適用場面**: ダッシュボード、レポーティング、キャッシュ代替。決済・在庫確認など整合性必須のケースには使わない。

### 2.2 トランザクションのベストプラクティス

**ミューテーション制限**:
| 制限 | 値 |
|------|---|
| 1コミットあたり最大ミューテーション数 | **80,000** |
| トランザクションサイズ上限 | **100 MB** |

**ミューテーション数の計算**:
- INSERT: 影響カラム数（PKカラム含む）
- UPDATE: PKカラム数 + 更新カラム数
- DELETE: 1行 = 1（+ セカンダリインデックス分）

**ロック競合の防止**:
- Read-Writeトランザクション内では **必要最小限のキー範囲** のみ読み取る
- トランザクション内でリモートAPI呼び出し・複雑なビジネスロジックを実行しない
- **読み取りだけなら必ず読み取り専用トランザクションを使う**（ロック不要）

**リトライ**: 初期遅延 500ms、最大遅延 16秒、スケール係数 1.5倍（ジッター付き）。同一セッション内リトライでロック優先度が上昇。

### 2.3 読み書きトランザクション vs 読み取り専用トランザクション

| 特性 | 読み書き | 読み取り専用 |
|------|---------|------------|
| ロック取得 | あり | **なし** |
| アボートの可能性 | あり | なし |
| リーダーレプリカ通信 | 必須 | Strong時のみ |
| 並行書き込みのブロック | あり | **なし** |

**運用ルール（ColorSing）**: RW 内で `SELECT` したレコードには**共有ロック（shared read lock — 他トランザクションの読み取りは可、書き込みはブロック）**がかかるため、書き込みが不要な参照は必ず RO で取得すること。RW のスコープを最小化し、書き込み直前に必要なキーのみ読むようにする。詳細は **8.1 初級編 最低限ルール** の「RW を狭く、RO を優先」を参照。

### 2.4 Partitioned DML

大規模な更新・削除に使用。自動的にパーティション分割して独立トランザクションとして実行。

**制限**:
- Partitioned DML は内部で複数のパーティション単位トランザクションに分割され、**各パーティションのトランザクションがそれぞれ 80,000 ミューテーションの制限に従う**（全体としての「1 コミット 80,000」という制約ではない）
- 同時実行数: 最大 **20,000**（100超は内部ロック競合リスク）
- **冪等性が必須**（リトライで同一パーティションが複数回実行される可能性）

```sql
-- 危険: 冪等ではない
UPDATE orders SET retry_count = retry_count + 1 WHERE status = 'FAILED';

-- 安全: 冪等
UPDATE orders SET status = 'ARCHIVED' WHERE created_at < '2023-01-01';
```

**親テーブル削除時の注意**: インターリーブ子テーブルのグローバルインデックスエントリもミューテーションにカウント。先に子→親の順で削除。

### 2.5 クエリオプティマイザ

- 本番では **OPTIMIZER_VERSION を固定** し、バージョン変更時に事前テスト
- `FORCE_INDEX` ヒントで明示的にインデックス指定（自動選択は限定的）
- 統計情報は **3日ごと** に自動更新。大量インポート後は `ANALYZE` で手動更新（反映まで最大10分）

```sql
SELECT s.SingerId, s.FirstName, s.SingerInfo
FROM Singers@{FORCE_INDEX=SingersByFirstLastName}
WHERE FirstName = 'Catalina' AND LastName = 'Smith';
```

### 2.6 クエリパラメータバインディング

**必ずパラメータバインディングを使う**。リテラル埋め込みは毎回コンパイル、パラメータバインディングはプランキャッシュを再利用。

```go
// 悪い例: プランキャッシュが効かない
stmt := spanner.Statement{
    SQL: fmt.Sprintf("SELECT * FROM Users WHERE id = %d", userID),
}

// 良い例: プランキャッシュが有効 + SQLインジェクション防止
stmt := spanner.Statement{
    SQL:    "SELECT * FROM Users WHERE id = @id",
    Params: map[string]interface{}{"id": userID},
}
```

### 2.7 ミューテーション vs DML

| 比較軸 | ミューテーション | DML |
|--------|-----------------|-----|
| レイテンシ | 低い（約1/4） | 高い |
| バッファリング | クライアント側でバッファ、コミット時一括 | 各ステートメントごとにサーバー往復 |
| 書き込み後の読み取り | 不可 | 可 |
| 制約チェック | コミット時一括 | 即時 |

**同一トランザクション内でDMLとミューテーションを混在させない。**

## 3. 運用・スケーリング & キャパシティプランニング

### 3.1 Processing Units vs Nodes

```
1 Node = 1,000 Processing Units (PU)
```

| インスタンスサイズ | スケール粒度 |
|------------------|------------|
| 100〜900 PU | **100 PU 単位** |
| 1,000 PU 以上 | **1,000 PU（1ノード）単位** |

スループット目安: 1ノードあたり最大 **2,000 QPS**（Point Read）、**1,000 QPS**（複雑クエリ）。

### 3.2 Autoscaler

**推奨設定値**:
- リージョナル: `highPriorityCpu` = 65%、`totalCpu` = 70%
- マルチリージョン: `highPriorityCpu` = 45%、`totalCpu` = 50%

**非対称スケーリング**: スケールアップは迅速、スケールダウンは段階的（デフォルトステップ 2,000 PU）。スプリット再配置の時間を確保するため。

### 3.3 マルチリージョン構成

| 構成名 | カバーリージョン | SLA |
|--------|---------------|-----|
| リージョナル | 1リージョン | **99.99%** |
| `nam6` 等 | 複数リージョン | **99.999%** |

**リーダーリージョンの重要性**: 書き込みはリーダー経由でクォーラムを形成するため、アプリとリーダーの距離が書き込みレイテンシに直結。

| 操作 | 単一リージョン | マルチリージョン（同大陸） | マルチリージョン（異大陸） |
|------|-------------|------------------------|------------------------|
| 書き込み | 数ms | 10〜20ms | 60ms〜 |
| Strong Read | 数ms | 数ms（リーダー近傍） | 高い |
| Stale Read ≥15秒 | 数ms | **数ms（どこからでも）** | 数ms |

**コスト**: マルチリージョンはリージョナルの約 **3〜8倍**。

### 3.4 ホットスポット検出 — Key Visualizer

| パターン | 見た目 | 原因 | 対処 |
|---------|--------|------|------|
| 対角線 | 斜めの明るい線 | 連番キーへの順次書き込み | UUID / Bit-Reversed |
| 水平バンド | 特定キーレンジが明るい帯 | 特定行への集中アクセス | スキーマ再設計 |
| 均一な明るさ | 全体的に同じ色 | **理想的な分散** | そのまま |

**Pre-Splitting**: 大量データ投入前に **1ノードあたり10スプリットポイント** を事前設定。

### 3.5 バックアップ & PITR

| 種類 | 推奨頻度（運用目安） | 最大保持期間 |
|------|---------------------|------------|
| フルバックアップ | 12時間ごと（推奨値、公式仕様ではない） | **1年** |
| 増分バックアップ | 4時間ごと（推奨値、公式仕様ではない） | **1年** |
| PITR | - | **最大7日**（マイクロ秒精度） |

```sql
-- 過去のデータを読み取る（誤削除からのリカバリ）
SELECT * FROM Orders
  AS OF SYSTEM TIME '2026-04-10T10:00:00Z'
WHERE order_id = '12345';
```

**重要**: `version_retention_period` はデフォルト最小値（1時間）→ 本番では **7日** に延長設定。PITR は独立した課金単価（例: `$0.50/GB/月`）があるわけではなく、`version_retention_period` を延長することで旧バージョンの行が DB ストレージ（リージョナル `$0.30/GB/月` 相当）として課金される構造。保持期間を伸ばすほどストレージ使用量が増える点に注意。

### 3.6 Change Streams（CDC）

- 保持期間: 最大 **7日**（推奨: 3〜7日）
- Dataflow連携テンプレート: BigQuery / Pub/Sub / GCS
- Co-location設計: 変更レコードをデータ変更と同一トランザクションで書き込み、オーバーヘッド最小化
- **保持期間デフォルト1日のまま運用すると、パイプライン1日以上停止でデータロス**

### 3.7 Export/Import

- Dataflow + Avro フォーマット推奨
- 同一リージョンにSpanner・GCSバケット・Dataflowジョブを配置
- **セカンダリインデックスはインポート後に作成**
- **外部キー・インターリーブは事前削除、インポート後に再作成**

### 3.8 メンテナンス

- メンテナンスウィンドウの設定は **不要**（Google が自動管理）
- スキーマ変更はオンライン実行、読み書き継続可能
- クライアントライブラリのバージョンアップは別途必要

## 4. 監視・アラート & トラブルシューティング

### 4.1 主要メトリクス

| メトリクス | 説明 | 警戒水準 |
|----------|------|---------|
| `instance/cpu/utilization` | CPU使用率（全優先度） | Regional: 65%, Multi-region: 45% |
| `instance/cpu/utilization_by_priority` | 優先度別CPU | HIGH が 65% 超で要調査 |
| `instance/cpu/smoothed_utilization` | 24時間平均CPU | 長期トレンド・アラートに使用 |
| `instance/storage/used_bytes` | ストレージ使用量 | 上限の70%で警告 |
| `api/request_latencies` | APIレイテンシ P50/P99/P999 | ワークロード依存 |
| `lock/wait_time` | ロック待機時間 | P99急増で要調査 |

### 4.2 CPU使用率 — なぜ65%が推奨か

1. **フェイルオーバーヘッドルーム**: コンポーネント障害時に残りが引き受けるためのバッファ
2. **ローリングリスタート耐性**: メンテナンス中の一時的なキャパシティ減少に対応
3. **SLA保証条件**: 65%超の継続運用はSLA対象外リスク

**High Priority CPU vs Total CPU**: High Priority = ユーザーリクエスト、System = コンパクション等。Total CPUが常時高いとシステムタスクが後回しにされ最終的にユーザーワークロードに干渉。

### 4.3 SPANNER_SYS システムテーブル

| テーブルグループ | 粒度 | 保持期間 |
|---------------|------|---------|
| `QUERY_STATS_TOP_MINUTE` | 1分 | 6時間 |
| `QUERY_STATS_TOP_10MINUTE` | 10分 | 4日 |
| `QUERY_STATS_TOP_HOUR` | 1時間 | 30日 |
| `TXN_STATS_TOP_*` | 同上 | 同上 |
| `LOCK_STATS_TOP_*` | 同上 | 同上 |

**実践的クエリ**:

```sql
-- CPU消費上位クエリ（直近1時間）
SELECT text, avg_cpu_seconds, avg_latency_seconds, execution_count,
       avg_rows_scanned, avg_rows_scanned / NULLIF(avg_rows_returned, 0) AS scan_ratio
FROM SPANNER_SYS.QUERY_STATS_TOP_HOUR
WHERE interval_end = (SELECT MAX(interval_end) FROM SPANNER_SYS.QUERY_STATS_TOP_HOUR)
ORDER BY avg_cpu_seconds DESC LIMIT 10;

-- ロック競合が多いキー範囲（直近1時間）
SELECT row_range_start_key, total_lock_wait_seconds, sample_lock_requests
FROM SPANNER_SYS.LOCK_STATS_TOP_HOUR
ORDER BY total_lock_wait_seconds DESC LIMIT 10;

-- アボート率が高いトランザクション
SELECT transaction_tag, commit_attempt_count, commit_abort_count,
       SAFE_DIVIDE(commit_abort_count, commit_attempt_count) AS abort_rate
FROM SPANNER_SYS.TXN_STATS_TOP_10MINUTE
ORDER BY abort_rate DESC LIMIT 10;
```

**トランザクションタグ・リクエストタグ**: 本番では必ず付与して `SPANNER_SYS` での絞り込みを容易にする。

### 4.4 アラート設計

**必須アラート**:

| アラート | 条件 | 理由 |
|---------|------|------|
| CPU（High Priority） | > 65% が5分継続 | フェイルオーバーマージン |
| Smoothed CPU | > 60% が24時間継続 | キャパシティプランニング |
| API レイテンシ P99 | SLO閾値の80%超え | バーンレート予兆 |
| エラーレート | UNAVAILABLE/DEADLINE_EXCEEDED > 1% | サービス影響 |
| ストレージ | 上限の70%超え | 容量計画 |

**アラートすべきでないもの**: 一時的CPUスパイク（1〜2分）、Low Priority CPU高騰単独、少量のNOT_FOUNDエラー。

### 4.5 よくある障害パターン

**DEADLINE_EXCEEDED**:
1. CPU過負荷 → `QUERY_STATS_TOP_MINUTE` で重量クエリ特定
2. 非効率クエリ → `avg_rows_scanned` >> `avg_rows_returned` のクエリを探す
3. タイムアウト設定 → **短すぎるタイムアウト + 即リトライはアンチパターン**

**ABORTED（トランザクション競合）**:
1. `TXN_STATS` でアボート率確認
2. `LOCK_STATS` でロック競合キー範囲特定
3. 対策: トランザクション保持時間最小化、読み取り専用トランザクション活用、楽観的ロック検討

**セッションプール枯渇**:
- `in_use_sessions` が `max_allowed_sessions` に張り付き → セッションリーク疑い
- Java: `setTrackStackTraceOfSessionCheckout(true)` で追跡

### 4.6 監査ログ

| ログ種別 | 対象 | デフォルト |
|---------|------|----------|
| Admin Activity | インスタンス/DB作成変更、IAM変更 | 常時有効（無効化不可） |
| Data Access | SQLクエリ、データ読み書き | **無効**（有効化が必要） |

Data Access Logsは大量に生成されコスト急増の可能性あり → 必要なサービスに限定して有効化。

## 5. クライアントライブラリパターン & 接続管理

> **注記**: 本セクションは公式 Go SDK（`cloud.google.com/go/spanner`）を前提にした汎用ノウハウ。ColorSing で使われている `transactable`（`BeginRwTransaction` / `BeginRoTransaction` / `BeginRwTransactionWithUserLock` / `AfterCommit` 等）は独自ラッパーで、挙動や API 名はここでの説明と異なる。ColorSing 固有の仕様は別スキルドキュメント（`cloud-spanner/` 配下）を参照。

### 5.1 セッション管理

| パラメータ | デフォルト | 推奨 |
|----------|----------|------|
| `MinSessions` | 100 | 予想される定常並列数 |
| `MaxSessions` | 400 | 最大同時トランザクション数 |
| `NumChannels` | 4 | `MaxSessions / 100` |

**Multiplexed Sessions**: 1 セッションで大量の並列リクエスト処理。keep-alive 不要、セッション管理簡略化。Read / Read-Only トランザクション向けは既定で有効化が進んでいる一方、Read-Write トランザクションへの適用は 2025 年以降段階的にロールアウト中。具体的なデフォルト化の状況・バージョンは使用中の SDK CHANGELOG（Go の場合は [google-cloud-go CHANGELOG](https://github.com/googleapis/google-cloud-go/blob/main/spanner/CHANGES.md)）で確認する。

```bash
export GOOGLE_CLOUD_SPANNER_MULTIPLEXED_SESSIONS=true
```

**セッションリーク検出**: `ListSessions` APIで最終使用時刻確認、OpenTelemetryでセッションプールメトリクス収集。

### 5.2 gRPC チャネルプール

- 1チャネルは **最大100並列リクエスト** を処理
- **Node.js は複数gRPCチャネル非対応** → 100セッション超なら複数クライアント作成

### 5.3 リトライ & エラーハンドリング

| エラーコード | リトライ | 説明 |
|------------|---------|------|
| `ABORTED` | **必須**（同一セッション内） | トランザクション競合 |
| `UNAVAILABLE` | 自動対象 | 一時的ネットワーク問題 |
| `DEADLINE_EXCEEDED` | 状況による | タイムアウト超過 |
| `NOT_FOUND` | 不可 | リソース不存在 |

**ABORTED はライブラリの `RunInTransaction` 内で自動リトライ。冪等性を確保すること。**

### 5.4 言語別の注意点

**Go**: `google-cloud-go/spanner` は Multiplexed Sessions の対応を段階的に拡大中。現時点でのデフォルト有効化状況は [CHANGELOG](https://github.com/googleapis/google-cloud-go/blob/main/spanner/CHANGES.md) を参照。

**Java**: `setWriteSessionsFraction()` は非推奨（効果なし）。Spring Bootではシングルトンクライアント必須。

**Python**: デフォルト `BurstyPool`（高負荷時にセッション爆発の可能性）→ `FixedSizePool` 推奨。

**Node.js**: 複数gRPCチャネル非対応。`database.getTransaction()` 手動使用時はリリース忘れに注意。

### 5.5 リクエストプライオリティ

| プライオリティ | 用途 |
|--------------|------|
| `PRIORITY_HIGH` | OLTP、ユーザー向けリアルタイム処理 |
| `PRIORITY_MEDIUM` | デフォルト |
| `PRIORITY_LOW` | バッチ処理、分析クエリ |

```go
// 分析クエリはLOWプライオリティ
iter := client.Single().QueryWithOptions(ctx, stmt,
    spanner.QueryOptions{Priority: sppb.RequestOptions_PRIORITY_LOW})
```

### 5.6 GKE からの接続

- **Workload Identity** を使用（サービスアカウントキー禁止）
- GKE メタデータサーバーの並列処理能力には上限があるため、ノード上で大量の認証リクエストが走ると輻輳しうる（具体的な同時接続数は公式に明示されていない。高負荷時はトークンキャッシュ・SDK のデフォルトリトライで吸収する前提で設計する）
- 各 Pod の `MaxSessions` × レプリカ数がセッション上限（10,000/DB）を超えないよう設計

### 5.7 エミュレータ

```bash

export SPANNER_EMULATOR_HOST="localhost:9010"
```

**制限**: 並列 read-write トランザクションには制限あり（DB 全体ロック相当の挙動で直列化されるケースがある）、メモリのみ（永続化なし）、SPANNER_SYS 一部非対応、パフォーマンステスト不可。

### 6.1 コスト構造

|------|-------------------|
| 1ノード（1,000 PU） | 約 $0.90/時間 |
| 100 PU | 約 $0.09/時間 |
| ストレージ | 約 $0.30/GB/月 |
| マルチリージョン | リージョナルの約3〜8倍 |

**エディション（2024年9月〜）**: Standard（基本機能、リージョナル/マルチリージョン構成とも選択可能）、Enterprise（Graph / Vector / 高度なオートスケーラー / フルテキスト検索などの追加機能）、Enterprise Plus（Enterprise の機能に加え高度なワークロード最適化・先進機能が付与）。いずれも 99.999% SLA はマルチリージョン構成で適用される。正確な機能差分は公式の [Spanner editions](https://cloud.google.com/spanner/docs/editions-overview) を参照。

### 6.2 コスト最適化戦略

| 施策 | 効果 |
|------|------|
| CUD 3年 | コンピュートコスト **最大40%削減** |
| CUD 1年 | コンピュートコスト **20%削減** |
| オートスケーラー | 閑散時の余剰コスト削減 |
| Stale Reads | 読み取り負荷軽減 |
| インターリーブ | 分散JOIN回避 |
| TTL | ストレージコスト削減 |
| 100 PU インスタンス | 開発環境を月額 ~$65 に |

**CUD はベースライン（常時必要な最小PU）のみに購入。オートスケーラーのmax設定を忘れずに。**

### 6.3 セキュリティ

**CMEK（顧客管理暗号鍵）**:
- DB **作成時** に指定（後から変更不可）
- KMSキーはSpannerと **同じロケーション** に配置
- キーを誤って無効化/削除すると **データアクセス完全喪失**

**VPC Service Controls**:
- 必ず **DryRunモード** から開始
- Spanner単体でなく連携サービス（BigQuery, GCS, Dataflow）も同一境界内に

**Fine-Grained Access Control（FGAC）**:

```sql
CREATE ROLE inventory_reader;
GRANT SELECT ON TABLE Products TO ROLE inventory_reader;
GRANT SELECT (customer_name, order_id) ON TABLE Orders TO ROLE inventory_reader;
-- 機密カラム（credit_card_number）は除外
```

### 6.4 移行戦略

**Spanner Migration Tool（SMT）**: MySQL, PostgreSQL, SQL Server, Oracle, DynamoDBから移行支援。

**推奨移行フロー（最小ダウンタイム）**:

```
Phase 1: スキーマ変換・初期ロード
Phase 2: CDCで増分同期開始
Phase 3: デュアルライトモード（書き込み: ソース+Spanner、読み取り: ソース）
Phase 4: DVT（Data Validation Tool）で整合性検証
Phase 5: カットオーバー（Spanner完全移行）
Phase 6: リバースレプリケーション維持（フォールバック用）
```

**最大の落とし穴**: 連番主キー（AUTO_INCREMENT）→ UUID v4 / ビット反転シーケンスへの変換が必須。

### 6.5 PostgreSQL インターフェース

- DB作成時にダイアレクト選択（**後から変更不可**）
- DDLトランザクション非サポート
- 多くのPostgreSQL組み込み関数が未サポート
- **GoogleSQLが推奨**（移行の便宜以外では優位性なし）

### 6.6 統合パターン

| 統合先 | パターン | ユースケース |
|--------|---------|-------------|
| BigQuery | Change Streams + Dataflow | リアルタイム分析 |
| BigQuery | フェデレーテッドクエリ | 少量のリアルタイムルックアップ |
| Pub/Sub | Change Streams + Dataflow | イベント駆動アーキテクチャ |

## 8. ColorSing 運用ノウハウ

> ColorSing サービスで蓄積された Spanner 実装・運用知見。過去の障害・PR レビューから抽出されたルール。

### 8.1 初級編 最低限ルール（実装時に絶対に守る）

#### ルール1: RW トランザクションを狭く、RO を優先する

Spanner は RW で `SELECT` したレコードに**共有ロック**を取る。共有ロックの範囲を狭めないと他トランザクションが abort しやすくなる。

- **読み取りのみなら RO を使う**（ロック不要、abort なし、リトライ不要）
- **RO で取ったデータの注意点**:
  - データが少し古いことがある → 最新性が必要なら RW
  - 取得データをロックできない → 排他したいなら RW
- **RW で SELECT するのは書き込みに必要なキーだけに絞る**

**悪い例**（ライブの参照だけなのに RW に入れている）:

```go
err := i.db.BeginRwTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    // (NG) ライブレコードのロックを取ってしまっている
    live, err = i.liveRepository.Find(ctx, txn, liveID)
    if err != nil { return err }

    liveComment = models.NewLiveComment(live.liveID, userID, mentionUserID, []byte(body))
    return i.liveCommentRepository.Create(ctx, txn, liveComment)
})
```

**良い例**:

```go
var live *models.Live
err := i.db.BeginRoTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    live, err = i.liveRepository.Find(ctx, txn, liveID) // (OK) RO
    return err
})

err = i.db.BeginRwTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    liveComment = models.NewLiveComment(live.liveID, userID, mentionUserID, []byte(body))
    return i.liveCommentRepository.Create(ctx, txn, liveComment)
})
```

#### ルール2: テーブルに適切な TTL を設定する

Spanner でもレコード数が多いほどクエリが遅くなる。アプリから参照される履歴系テーブルは TTL 必須。

| ケース | TTL |
|--------|-----|
| アプリ側で CREATE / DELETE を適切に行うテーブル（例: `OnAiredLives`） | 不要 |
| アプリ側で READ しない履歴テーブル（例: `UserCoinBalanceInOutHistories`） | 不要 |
| アプリ側で READ するが DELETE しないテーブル | **必要**（READ が重くならないように自動削除） |

```sql
-- 有効期限1日の TTL（厳密に1日で消えるわけではなく、Spanner が任意のタイミングで消す）
ALTER TABLE PubsubMessages ADD ROW DELETION POLICY (OLDER_THAN(CreatedAt, INTERVAL 1 DAY));
```

無限遡及が仕様上必要なときは、PdM と調整して遡及期間に上限を設ける（例: 1ヶ月、1年）。

#### ルール3: RW トランザクション内で外部サービスへの書き込み・publish をしない

Spanner + Go では **RW が abort されてリトライされる**ことが MySQL より高頻度で起こる。トランザクション内で Redis / Pub/Sub / NATS などに書き込むと、以下の事故が発生する。

- Spanner は rollback したが外部には書き込み済み → データ不整合
- リトライされて外部への通知が重複発火

**対策**: トランザクション終了後に実行するか、`AfterCommit` フックを使う。

**悪い例**:

```go
err := i.db.BeginRwTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    if err := i.followingRepository.Create(ctx, txn, following); err != nil { return err }
    // (NG) rollback されても通知されてしまう / リトライで重複通知
    return i.publishFollowEventOnFirstTime(ctx, txn, ...)
})
```

**良い例1**（トランザクション外で publish）:

```go
err := i.db.BeginRwTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    return i.followingRepository.Create(ctx, txn, following)
})
if err != nil { return err }
err = i.publishFollowEventOnFirstTime(ctx, ...) // (OK) 1回だけ実行される
```

**良い例2**（AfterCommit フック）:

```go
err := i.db.BeginRwTransaction(ctx, func(ctx context.Context, txn transactable.Transaction) error {
    if err := i.followingRepository.Create(ctx, txn, following); err != nil { return err }
    // (OK) commit 成功後にのみ実行される
    return txn.AfterCommit(func(ctx context.Context) error {
        return i.publishFollowEventOnFirstTime(ctx, ...)
    })
})
```

#### ルール4: 必要に応じて悲観ロックを取る（Redis ベース）

Spanner は**楽観ロック方式**。複数トランザクションが同じレコードを競合するとどちらかが abort → Go SDK がリトライ → 最終的に失敗することもある。

**悲観ロックが必要なケース（ColorSing 実例）**:
- 複数リスナーが同じライバー情報を読み書きする
- 複数リスナーが同じ配信情報を読み書きする

ColorSing では **Redis を使った悲観ロック** を実装している（`BeginRwTransactionWithUserLock`）。同じキーに対するトランザクションはロック解放まで待機する。

```go
err = i.db.BeginRwTransactionWithUserLock(ctx, i.redisClient, liverUserID,
    func(ctx context.Context, txn transactable.Transaction) error {
        // 同じ liverUserID へのトランザクションは直列化される
    })
```

### 8.2 中級編 設計時に意識すべきこと

#### ホットスポット回避（PK & セカンダリインデックス）

- PK 先頭は**とりあえず UUIDv4**（迷ったらこれ）
- 単調増加値（タイムスタンプ・連番・UUIDv7）を PK 先頭に置かない
- **セカンダリインデックスも通常テーブルと同等に扱われる** → インデックス先頭に単調増加値・ホットな値（`LiveId` など多数のユーザーが同時参照するキー）を置かない
- 特殊ケースでは論理シャード・ビット順逆転を使う

参考: [PKでホットスポットを避ける | Spanner](https://cloud.google.com/spanner/docs/schema-design?hl=ja#primary-key-prevent-hotspots)

#### 新規クエリは必ずクエリプランを確認する

Cloud Console でクエリプランを見て最低限チェック:

- **スキャン行数**: フルスキャンになっていないか
- **Distributed Union 演算子**: 複数ノードを横断するクエリになっていないか（レイテンシ悪化）

参考: [Query execution plans | Spanner](https://cloud.google.com/spanner/docs/query-execution-plans?hl=ja)

#### ロック競合時の優先度

ロックが競合すると、優先度に応じて片方が待機（wait）するか中断（abort）される。優先度は以下で決まる:

- トランザクションの最初のオペレーションが早い方が高い
- 一度リトライされたトランザクションは優先度が上がる
- 明示的に指定も可能（`PRIORITY_HIGH/MEDIUM/LOW`）

**注意**: abort されたトランザクションは SDK が自動リトライする。**トランザクション外の slice に append するような副作用**がリトライで二重実行されないよう、副作用は `AfterCommit` か外部に出す。

### 8.3 インターリーブの設計指針

ColorSing のアーキテクチャ解説（Notion）より要点:

- Split は最初 1 つしかなく、負荷 / サイズ（約 8GB 上限）で動的分割
- ノード数を**徐々に増やさない**と Split 分散前で性能が出ない
- ノード数の**急減**は abort 率上昇 → 全体ノード数に対する一定割合ずつ減らす
- **Split をまたぐクエリは Distributed Union でレイテンシ悪化** → インターリーブで同一 Split に同居させる

「特定のエンティティに付随するデータは積極的にインターリーブ」が基本方針。ただし、インターリーブの制約（8GB 上限、1 親のみ、深度 7 レベル）を超える設計は避ける。

### 8.4 ページング

| 方式 | 使いどころ |
|------|----------|
| **Cursor-based** | キャッシュが難しいデータ。効率的 |
| **Offset-based** | スキャン結果を一定期間キャッシュする前提のみ。OFFSET 行もスキャンされ、各ノードで `m+n` 行スキャン後に結合するため分散 DB では特にコスト大 |

**Cursor-based の実装例**（全ユーザーの投稿を新しい順にページング）:

```sql
SELECT * FROM UserPosts
WHERE
  CreatedAt < @PreviousCreatedAt
  OR (CreatedAt = @PreviousCreatedAt AND UserId > @PreviousUserId)
ORDER BY CreatedAt DESC, UserId ASC
LIMIT @Limit;
```

同一時刻の衝突に備えてソート安定化用の列（上記では `UserId`）を追加。

### 8.5 FORCE_INDEX の使いどころ

- **基本的に常に `FORCE_INDEX` を指定する**。Google 管理のオプティマイザが過去にインデックスを急に使わなくなり性能劣化した実例がある
- **ただしカバリングインデックスでない場合、大量レコードのスキャンでは避ける**。インデックスに含まれないカラムのために結合処理が走ると逆に遅い
- 大量スキャンで結合が必要になるなら: スキャン範囲を絞る or `STORING` でカバリング化

```sql
-- 基本形
SELECT * FROM Users@{FORCE_INDEX=UsersIndexHoge} WHERE ~

-- NG: カバリングではない + 大量スキャン
SELECT UserId, Name, Age
FROM Users@{FORCE_INDEX=UsersByAge}  -- Age だけのインデックス
WHERE Age = 20;  -- Name を取るためにテーブル結合が発生
```

### 8.6 バッチ処理の負荷分散

全ユーザー対象のバッチを作成するとき:

- PK の先頭（UUIDv4）の**先頭 1 桁** で 16 分割して並列実行
- 並列プロセス同士が競合しないかは**設計段階で必ず確認**する（書き込み対象の重複がないか）

### 8.7 カラム追加の制約（MySQL との差分）

Spanner の ALTER TABLE は MySQL と異なる制約がある。

- **`ADD COLUMN ... NOT NULL` は不可** → 2 ステップで対応（Spanner GoogleSQL は `MODIFY COLUMN` ではなく `ALTER COLUMN` を使う。また識別子にバッククォートは不要）

  ```sql
  ALTER TABLE Users ADD COLUMN Age INT64 DEFAULT (0);
  ALTER TABLE Users ALTER COLUMN Age INT64 NOT NULL;
  ```

- **`AFTER` 句は不可** → カラムは常にテーブル末尾に追加される
- バックフィルが発生する `NOT NULL` 制約化・インデックス追加は、実行中でも NULL 書き込みがほぼ即座に拒否されるので計画的に

### 8.8 スキーマ管理

ColorSing ではサーバーアプリケーションのリポジトリ内にスキーマディレクトリを作り、アプリ変更と紐づけて管理している。

| 形式 | メリット | デメリット |
|------|---------|-----------|
| CREATE TABLE 形式で完全な DDL を定義 | テーブル定義がファイルから見やすい | マイグレーションの一連のワークフローが定義しづらい |
| **ALTER TABLE で差分を別ファイルに定義（推奨）** | 時系列順に管理、既存マイグレーションツール（yo 等）と連携可能 | 最新テーブル定義の見通しが悪い |

ColorSing では **差分形式（ALTER 差分 + yo などのマイグレーションツール）** を採用している。

### 8.9 クエリビルダとモデル生成（Go）

ColorSing では `database/sql` パッケージで Spanner に接続。クエリ組み立てはケースごとに条件が変わるので、手組みせずライブラリを使う。

| ツール | 種類 | 選定結果 |
|--------|------|----------|
| **memeduck** | Spanner 用クエリビルダ | **採用**。JOIN はサポートされていないが、インターリーブ子テーブルはサブクエリで取る方がパフォーマンスが良いので制約を逆手に取れる |
| spansql-query-builder | Spanner 用クエリビルダ | 検証したが非採用 |
| GORM / goqu | 汎用 ORM / クエリビルダ | Spanner 非公式サポートのため見送り |
| **mercari/yo** | モデル自動生成ツール | 使用。テーブル定義からモデル生成、テンプレートで共通クエリ生成。モデルにカスタムメソッドを生やしづらいのが難点 |

### 8.10 オートスケーリング運用

Spanner の Managed Autoscaler は CPU 使用率（高優先度・全体）やストレージ利用で min/max ノード数の範囲で自動調整する。ColorSing ではイベント時に高めの下限を設定している。

**即時スケールアウトの手順**（緊急時）:

1. GCP Console → Spanner インスタンス概要
2. 「インスタンスを編集」→「コンピューティング容量を構成する」→「スケーリングモード」→「**手動で割り当てる**」
3. 数量を上げて保存

ノード数を急に下げると abort 率が上がる（Split のリバランス中）ので、一定割合ずつ下げる。サービスごとに適切な割合を見積もること。

### 8.11 バックアップ方針（ColorSing）

#### 多層防御アーキテクチャ

PITR とネイティブバックアップは**同一 GCP プロジェクト内**にあるため、プロジェクト権限侵害（フィッシング、SA キー漏洩、内部不正）で同時に無効化される。**エアギャップバックアップが必須**。

ColorSing では以下の 4 層構成を採用:

| 層 | 手段 | 保存先 | 保持 | エアギャップ | 対応する脅威 |
|----|------|--------|------|------------|------------|
| **Layer 0** | PITR | Spanner 内部（同 PJ） | 7日 | なし | オペミス、アプリバグ |
| **Layer 1** | 日次フルバックアップ | Spanner マネージド（同 PJ） | 3日 | なし | 大規模データ破損、PITR フォールバック |
| **Layer 2** | 週次 Tier1 エクスポート | **別 PJ GCS（Nearline + WORM）** | 90日 | **あり** | ランサムウェア、権限侵害、内部不正 |
| **Layer 3** | 月次フルエクスポート | **別 PJ GCS（Coldline + WORM）** | 365日 | **あり** | 長期潜伏 APT、法的要請、監査対応 |

月額コスト目安: 合計 約 $235/月（894GB 本番 DB 規模）

#### RPO/RTO 目標

| シナリオ | RPO | RTO | 使用手段 | 復元方式 |
|---------|-----|-----|---------|---------|
| オペミス（誤 DML 等） | 0分 | 数分 | Layer 0: PITR | DB 全体復元 |
| 大規模データ破損 | 最大24時間 | 30分〜2時間 | Layer 1: ネイティブ BK | DB 全体復元 |
| ランサムウェア / 権限侵害 | 最大7日 | 数時間 | Layer 2: 週次エアギャップ | **テーブル単位の部分復元** |
| 長期潜伏 APT / 監査対応 | 最大30日 | 半日〜1日 | Layer 3: 月次エアギャップ | テーブル単位 / フォレンジック |

#### テーブルのティア分類

全テーブル一律ではなく、Layer 2（週次エアギャップ）の対象を絞る。

- **Tier 1 クリティカル**: 金銭・残高系（`UserCoinBalances`, `UserDiamondBalances`, `WithdrawalDiamondLedger`, `UserBankAccounts`, `ListenerSendGiftHistories` など）
- **Tier 2 重要**: ユーザー基盤（`Users`, `UserProfiles`, `Lives`, `OpeAuditLogs`）
- **Tier 3 再生成可能**: BQ 同期バッファ、TTL 7日前後の一時データ → Layer 2 から除外可

#### IAM / Organization Policy の要点

- **書き込み専用 SA**: `roles/storage.objectCreator` のみ、削除権限なし
- **別プロジェクト管理者**: singcolor PJ の管理者とは別の人物・グループ
- **Bucket Lock (WORM)**: `constraints/storage.retentionPolicyNotLocked = DENY`
- **Log Sink で監査ログも別 PJ に転送**: 攻撃者が同 PJ のログを消しても vault 側に証跡が残る
- **CMEK は導入しない**: キー削除でデータ復号不能化という新たな攻撃ベクトルを避ける（PCI-DSS Level 1 等になったら再検討）

#### ビジネス制約（重要）

**1 日以上前への全データ復元はビジネス判断としてほぼ不可能**。ライブ配信中のリアルタイム取引（ギフト、コイン、ダイヤモンド換金）が消失するため。
したがって復旧設計は **テーブル単位・行単位の部分復元**を前提にする。例: `UserCoinBalances` だけ過去状態に戻し、その後の入出金ログから再適用。

### 8.12 ColorSing 実装チェックリスト（PR レビュー観点）

Spanner 絡みの PR をレビューするとき、以下をチェック:

```
トランザクション
  [ ] 書き込み不要なのに RW を使っていないか
  [ ] RW のスコープが必要最小限か（書き込みに必要なキーのみ SELECT）
  [ ] RW 内で Redis / Pub/Sub / NATS / 外部 API を叩いていないか
  [ ] 外部サイドエフェクトは AfterCommit または TX 外に出ているか
  [ ] 競合必至のキー（ライバー・配信情報）には BeginRwTransactionWithUserLock を使っているか

スキーマ
  [ ] PK 先頭に単調増加値を置いていないか（UUIDv4 または bit-reversed）
  [ ] セカンダリインデックス先頭にホットキー（LiveId 等）を置いていないか
  [ ] 参照される履歴テーブルに TTL が設定されているか
  [ ] 新規 NOT NULL カラムは 2 段階 DDL になっているか
  [ ] インターリーブの 8GB 上限・深度 7 制約を踏んでいないか

クエリ
  [ ] 新規クエリのクエリプランを確認したか（フルスキャン・Distributed Union）
  [ ] FORCE_INDEX を指定しているか（カバリングでない場合の大量スキャンを除く）
  [ ] パラメータバインディングを使っているか（プランキャッシュ効く）
  [ ] ページングは Cursor-based になっているか（Offset-based はキャッシュ前提時のみ）

バッチ
  [ ] PK 先頭で分割した並列実行が競合を起こさないか
```
