> Google Cloud Spanner のスキーマ設計・クエリ最適化・運用・監視・クライアントパターン・コスト・セキュリティに関する実践的ノウハウ集

## 1. スキーマ設計 & データモデリング

### 1.1 プライマリキー設計

**鉄則: 単調増加キーを絶対に避ける。**

Spanner はデータをキーレンジ単位の「スプリット」に分割し、各スプリットを異なるノードに割り当てる。タイムスタンプや連番を先頭キーにすると、すべての新規書き込みがキー空間の末尾に集中する（**Moving Hotspot** 問題）。

| アプローチ | 構文例（GoogleSQL） | 特徴 |
|------------|---------------------|------|
| UUID v4 | `FanId UUID DEFAULT (NEW_UUID())` | 最もシンプル。衝突確率は 2^122 分の1 |
| ビット反転シーケンス | `GENERATED BY DEFAULT AS IDENTITY (BIT_REVERSED_POSITIVE)` | 整数IDが必要な場合。既存DBからの移行に適する |
| ハッシュプレフィックス | `FARM_FINGERPRINT(ActualKey) % N AS ShardId` | Nをノード数に合わせて設定 |

**定量的閾値**: 1スプリットが処理できる書き込みは約 **3,500 writes/秒**。超える見込みがあれば必ず分散キーを選ぶ。

**落とし穴**:
- **UUIDv7 は危険**: タイムスタンププレフィックスを持つため、単調増加キーと同等の問題を引き起こす
- **シャード数 N は静的**: `FARM_FINGERPRINT % N` のNを後から変えるにはデータ再書き込みが必要

```sql
-- BAD: 連番主キー
CREATE TABLE Bad (Id INT64 NOT NULL) PRIMARY KEY (Id);

-- GOOD: UUID v4
CREATE TABLE Good (Id UUID NOT NULL DEFAULT (NEW_UUID())) PRIMARY KEY (Id);

-- GOOD: ビット反転シーケンス
CREATE TABLE Better (
  Id INT64 NOT NULL GENERATED BY DEFAULT AS IDENTITY (BIT_REVERSED_POSITIVE)
) PRIMARY KEY (Id);

-- GOOD: ハッシュプレフィックス + タイムスタンプ
CREATE TABLE TimeSeries (
  ShardId INT64 NOT NULL,
  Timestamp TIMESTAMP NOT NULL,
  UserId STRING(36) NOT NULL,
) PRIMARY KEY (ShardId, Timestamp DESC);
```

### 1.2 インターリーブテーブル

親子関係のデータを同一スプリットに物理的に同居させ、分散JOINを排除する。

```sql
CREATE TABLE Singers (
  SingerId INT64 NOT NULL,
  Name STRING(1024),
) PRIMARY KEY (SingerId);

CREATE TABLE Albums (
  SingerId INT64 NOT NULL,
  AlbumId INT64 NOT NULL,
  Title STRING(1024),
) PRIMARY KEY (SingerId, AlbumId),
  INTERLEAVE IN PARENT Singers ON DELETE CASCADE;
```

**使うべき場面**:
- 親と子を一緒に読む操作が全体の80%以上
- 書き込み時に親子を同一トランザクションでまとめて書く
- 子テーブルの主要アクセスが「特定の親に属する子一覧」

**避けるべき場面**:
- 全親にわたって子テーブルを水平スキャン（逆に遅くなる）
- 親1件に対して子行が数万〜数百万行（親+子合計 ~8GB 超）
- 多対多関係（インターリーブは1親のみ）

**定量的効果**:
- 親子JOIN P99レイテンシ: **3ms** vs なし **15ms**（5倍改善）
- 書き込みレイテンシ: 7ms → 3ms（2PC参加者 2→1）

**制約**:
- 最大深度 **7レベル**（根テーブルを含む）
- TTLを使う親テーブルのインターリーブ子は **ON DELETE CASCADE** が必須

### 1.3 スキーマ変更（DDL）

スキーマ変更はダウンタイムゼロで実行可能だが、種類により所要時間が異なる。

| 操作 | 所要時間 | バックフィル |
|------|---------|-------------|
| 列追加（NULL許可） | 数分 | なし |
| テーブル + インデックス同時作成 | 数分 | なし |
| 既存テーブルへのインデックス追加 | 数分〜数時間 | あり |
| NOT NULL 制約追加 | 数分〜数時間 | あり（全行スキャン） |
| 外部キー追加（enforced） | 長時間 | あり |

**運用ルール**:
1. バックフィルが必要なインデックスは **1日3本まで** が推奨
2. テーブルとインデックスは **同一バッチで同時に作成**（別々だと2スキーマバージョンが生成）
3. **インデックス作成前にノードを増設**（開始後の増設は並列度に反映されない）
4. **バルクロード後にインデックスを作成**（ロード中のインデックス更新を回避）
5. スキーマオブジェクト上限: テーブル 5,000、インデックス 10,000、カラム 1,024/テーブル、PK列 16

**NOT NULL 追加の落とし穴**: バリデーション完了前でもNULL書き込みがほぼ即座に拒否される。

### 1.4 カラム型の選択

**NUMERIC vs FLOAT64**: 金融・会計データには必ず NUMERIC を使う。

| | NUMERIC | FLOAT64 |
|--|---------|---------|
| 0.2の格納 | 正確に 0.2 | 0.20000000000000001 |
| 精度 | 38桁、小数点以下9桁 | 64ビット浮動小数点（近似値） |

**JSON カラム**:
- インデックス不可 → 生成カラムで値を抽出してインデックス化
- ネスト深度上限: 80レベル
- コアフィールドはスキーマカラム化、可変データのみJSONに（ハイブリッドアプローチ）

**ARRAY カラム**:
- `ARRAY<ARRAY<T>>` は不可 → `ARRAY<STRUCT<field ARRAY<T>>>` で代替
- 子行が多い場合は別テーブル+インターリーブの方がスケーラブル

### 1.5 セカンダリインデックス

**STORING 句（カバリングインデックス）**: ベーステーブルへのバックジョインを回避。

```sql
CREATE INDEX AlbumsByTitle ON Albums(Title) STORING (ReleaseDate, MarketingBudget);
```

**NULL_FILTERED インデックス**: スパースカラムでサイズ大幅削減、書き込みオーバーヘッド低減。

```sql
CREATE NULL_FILTERED INDEX AlbumsByTitle ON Albums(Title);
```

**コミットタイムスタンプをインデックス先頭に置かない**:

```sql
-- 危険: ホットスポット発生
CREATE INDEX CommitIdx ON Events(CommitTimestamp);
-- 安全: 先頭に分散キー
CREATE INDEX CommitIdx ON Events(EventType, CommitTimestamp);
```

### 1.6 外部キー vs インターリーブ

| 観点 | インターリーブ | FK（Enforced） | FK（Informational） |
|------|------------|---------------------|--------------------------|
| 物理的データ共存 | あり | なし | なし |
| 参照整合性の強制 | 暗黙 | あり | なし |
| 書き込みオーバーヘッド | 低 | 高 | 低 |
| 複数親テーブル参照 | 不可 | 可 | 可 |

### 1.7 生成カラム（Generated Columns）

```sql
-- ケースインセンシティブ検索
ALTER TABLE Products ADD COLUMN UpperName STRING(MAX) AS (UPPER(Name)) STORED;
CREATE INDEX ProductsByUpperName ON Products(UpperName);

-- JSON フィールドのインデックス化
ALTER TABLE Events ADD COLUMN EventType STRING(100)
  AS (JSON_VALUE(Payload, '$.type')) STORED;
CREATE INDEX EventsByType ON Events(EventType);
```

**制約**: 同一テーブル内の列のみ参照可能、非決定論的関数は STORED 不可、式の変更不可（削除→再作成が必要）。

### 1.8 TTL（行削除ポリシー）

```sql
ALTER TABLE UserEvents
ADD ROW DELETION POLICY (OLDER_THAN(EventTime, INTERVAL 30 DAY));
```

**削除タイムライン**:
| 段階 | タイムライン |
|------|------------|
| TTL期限到来 | クエリには引き続き見える |
| 実際の削除実行 | **最大72時間以内** |
| ストレージ解放 | 削除から**最大7日以内** |

**トゥームストーン問題（最重要の落とし穴）**: 削除された行はバージョン保持期間中トゥームストーンとして残りスキャン対象になる。

実測例:
- 削除前の `SELECT MIN(ts)` → **16.56ms**
- 2,000万行削除後 → **7.09秒**

対策: クエリに `WHERE ts > [削除閾値]` を追加して削除済み範囲を除外。初回大量削除はPartitioned DMLで先行実施。

**その他の注意点**:
- テーブルあたり **1つのみ**
- 間隔は **整数日単位のみ**
- バックアップ復元後は **TTLポリシーが自動削除** → 手動再設定が必要
- Change Streamsで監査可能（`transaction_tag = 'RowDeletionPolicy'`）

## 2. クエリパフォーマンス & 最適化

### 2.1 ステール読み取り（Stale Reads）

| 種類 | 説明 | レイテンシ |
|------|------|----------|
| Strong Read | 常に最新データ | リーダーへのラウンドトリップ必須 |
| Bounded Staleness | 指定古さの上限内 | ネゴシエーション発生 |
| Exact Staleness | 指定タイムスタンプ | ネゴシエーション不要、最速 |

**推奨**: **15秒以上のExact Staleness**。リーダーは約10秒ごとにレプリカの安全タイムスタンプを更新するため、15秒ステールで近傍レプリカから直接読み取り可能。

**適用場面**: ダッシュボード、レポーティング、キャッシュ代替。決済・在庫確認など整合性必須のケースには使わない。

### 2.2 トランザクションのベストプラクティス

**ミューテーション制限**:
| 制限 | 値 |
|------|---|
| 1コミットあたり最大ミューテーション数 | **80,000** |
| トランザクションサイズ上限 | **100 MB** |

**ミューテーション数の計算**:
- INSERT: 影響カラム数（PKカラム含む）
- UPDATE: PKカラム数 + 更新カラム数
- DELETE: 1行 = 1（+ セカンダリインデックス分）

**ロック競合の防止**:
- Read-Writeトランザクション内では **必要最小限のキー範囲** のみ読み取る
- トランザクション内でリモートAPI呼び出し・複雑なビジネスロジックを実行しない
- **読み取りだけなら必ず読み取り専用トランザクションを使う**（ロック不要）

**リトライ**: 初期遅延 500ms、最大遅延 16秒、スケール係数 1.5倍（ジッター付き）。同一セッション内リトライでロック優先度が上昇。

### 2.3 読み書きトランザクション vs 読み取り専用トランザクション

| 特性 | 読み書き | 読み取り専用 |
|------|---------|------------|
| ロック取得 | あり | **なし** |
| アボートの可能性 | あり | なし |
| リーダーレプリカ通信 | 必須 | Strong時のみ |
| 並行書き込みのブロック | あり | **なし** |

### 2.4 Partitioned DML

大規模な更新・削除に使用。自動的にパーティション分割して独立トランザクションとして実行。

**制限**:
- 1コミットあたり最大 **80,000 ミューテーション**
- 同時実行数: 最大 **20,000**（100超は内部ロック競合リスク）
- **冪等性が必須**（リトライで同一パーティションが複数回実行される可能性）

```sql
-- 危険: 冪等ではない
UPDATE orders SET retry_count = retry_count + 1 WHERE status = 'FAILED';

-- 安全: 冪等
UPDATE orders SET status = 'ARCHIVED' WHERE created_at < '2023-01-01';
```

**親テーブル削除時の注意**: インターリーブ子テーブルのグローバルインデックスエントリもミューテーションにカウント。先に子→親の順で削除。

### 2.5 クエリオプティマイザ

- 本番では **OPTIMIZER_VERSION を固定** し、バージョン変更時に事前テスト
- `FORCE_INDEX` ヒントで明示的にインデックス指定（自動選択は限定的）
- 統計情報は **3日ごと** に自動更新。大量インポート後は `ANALYZE` で手動更新（反映まで最大10分）

```sql
SELECT s.SingerId, s.FirstName, s.SingerInfo
FROM Singers@{FORCE_INDEX=SingersByFirstLastName}
WHERE FirstName = 'Catalina' AND LastName = 'Smith';
```

### 2.6 クエリパラメータバインディング

**必ずパラメータバインディングを使う**。リテラル埋め込みは毎回コンパイル、パラメータバインディングはプランキャッシュを再利用。

```go
// 悪い例: プランキャッシュが効かない
stmt := spanner.Statement{
    SQL: fmt.Sprintf("SELECT * FROM Users WHERE id = %d", userID),
}

// 良い例: プランキャッシュが有効 + SQLインジェクション防止
stmt := spanner.Statement{
    SQL:    "SELECT * FROM Users WHERE id = @id",
    Params: map[string]interface{}{"id": userID},
}
```

### 2.7 ミューテーション vs DML

| 比較軸 | ミューテーション | DML |
|--------|-----------------|-----|
| レイテンシ | 低い（約1/4） | 高い |
| バッファリング | クライアント側でバッファ、コミット時一括 | 各ステートメントごとにサーバー往復 |
| 書き込み後の読み取り | 不可 | 可 |
| 制約チェック | コミット時一括 | 即時 |

**同一トランザクション内でDMLとミューテーションを混在させない。**

## 3. 運用・スケーリング & キャパシティプランニング

### 3.1 Processing Units vs Nodes

```
1 Node = 1,000 Processing Units (PU)
```

| インスタンスサイズ | スケール粒度 |
|------------------|------------|
| 100〜900 PU | **100 PU 単位** |
| 1,000 PU 以上 | **1,000 PU（1ノード）単位** |

スループット目安: 1ノードあたり最大 **2,000 QPS**（Point Read）、**1,000 QPS**（複雑クエリ）。

### 3.2 Autoscaler

**推奨設定値**:
- リージョナル: `highPriorityCpu` = 65%、`totalCpu` = 70%
- マルチリージョン: `highPriorityCpu` = 45%、`totalCpu` = 50%

**非対称スケーリング**: スケールアップは迅速、スケールダウンは段階的（デフォルトステップ 2,000 PU）。スプリット再配置の時間を確保するため。

**制約**: `minNodes` は `maxNodes` の **10%以上**（例: max=30 なら min=3 以上）。

### 3.3 マルチリージョン構成

| 構成名 | カバーリージョン | SLA |
|--------|---------------|-----|
| リージョナル | 1リージョン | **99.99%** |
| `nam6` 等 | 複数リージョン | **99.999%** |

**リーダーリージョンの重要性**: 書き込みはリーダー経由でクォーラムを形成するため、アプリとリーダーの距離が書き込みレイテンシに直結。

| 操作 | 単一リージョン | マルチリージョン（同大陸） | マルチリージョン（異大陸） |
|------|-------------|------------------------|------------------------|
| 書き込み | 数ms | 10〜20ms | 60ms〜 |
| Strong Read | 数ms | 数ms（リーダー近傍） | 高い |
| Stale Read ≥15秒 | 数ms | **数ms（どこからでも）** | 数ms |

**コスト**: マルチリージョンはリージョナルの約 **3〜8倍**。

### 3.4 ホットスポット検出 — Key Visualizer

| パターン | 見た目 | 原因 | 対処 |
|---------|--------|------|------|
| 対角線 | 斜めの明るい線 | 連番キーへの順次書き込み | UUID / Bit-Reversed |
| 水平バンド | 特定キーレンジが明るい帯 | 特定行への集中アクセス | スキーマ再設計 |
| 均一な明るさ | 全体的に同じ色 | **理想的な分散** | そのまま |

**Pre-Splitting**: 大量データ投入前に **1ノードあたり10スプリットポイント** を事前設定。

### 3.5 バックアップ & PITR

| 種類 | 最小頻度 | 最大保持期間 |
|------|---------|------------|
| フルバックアップ | 12時間ごと | **1年** |
| 増分バックアップ | 4時間ごと | **1年** |
| PITR | - | **最大7日**（マイクロ秒精度） |

```sql
-- 過去のデータを読み取る（誤削除からのリカバリ）
SELECT * FROM Orders
  AS OF SYSTEM TIME '2026-04-10T10:00:00Z'
WHERE order_id = '12345';
```

**重要**: `version_retention_period` はデフォルト最小値（1時間）→ 本番では **7日** に延長設定。

### 3.6 Change Streams（CDC）

- 保持期間: 最大 **7日**（推奨: 3〜7日）
- Dataflow連携テンプレート: BigQuery / Pub/Sub / GCS
- Co-location設計: 変更レコードをデータ変更と同一トランザクションで書き込み、オーバーヘッド最小化
- **保持期間デフォルト1日のまま運用すると、パイプライン1日以上停止でデータロス**

### 3.7 Export/Import

- Dataflow + Avro フォーマット推奨
- 同一リージョンにSpanner・GCSバケット・Dataflowジョブを配置
- **セカンダリインデックスはインポート後に作成**
- **外部キー・インターリーブは事前削除、インポート後に再作成**

### 3.8 メンテナンス

- メンテナンスウィンドウの設定は **不要**（Google が自動管理）
- スキーマ変更はオンライン実行、読み書き継続可能
- クライアントライブラリのバージョンアップは別途必要

## 4. 監視・アラート & トラブルシューティング

### 4.1 主要メトリクス

| メトリクス | 説明 | 警戒水準 |
|----------|------|---------|
| `instance/cpu/utilization` | CPU使用率（全優先度） | Regional: 65%, Multi-region: 45% |
| `instance/cpu/utilization_by_priority` | 優先度別CPU | HIGH が 65% 超で要調査 |
| `instance/cpu/smoothed_utilization` | 24時間平均CPU | 長期トレンド・アラートに使用 |
| `instance/storage/used_bytes` | ストレージ使用量 | 上限の70%で警告 |
| `api/request_latencies` | APIレイテンシ P50/P99/P999 | ワークロード依存 |
| `lock/wait_time` | ロック待機時間 | P99急増で要調査 |

### 4.2 CPU使用率 — なぜ65%が推奨か

1. **フェイルオーバーヘッドルーム**: コンポーネント障害時に残りが引き受けるためのバッファ
2. **ローリングリスタート耐性**: メンテナンス中の一時的なキャパシティ減少に対応
3. **SLA保証条件**: 65%超の継続運用はSLA対象外リスク

**High Priority CPU vs Total CPU**: High Priority = ユーザーリクエスト、System = コンパクション等。Total CPUが常時高いとシステムタスクが後回しにされ最終的にユーザーワークロードに干渉。

### 4.3 SPANNER_SYS システムテーブル

| テーブルグループ | 粒度 | 保持期間 |
|---------------|------|---------|
| `QUERY_STATS_TOP_MINUTE` | 1分 | 6時間 |
| `QUERY_STATS_TOP_10MINUTE` | 10分 | 4日 |
| `QUERY_STATS_TOP_HOUR` | 1時間 | 30日 |
| `TXN_STATS_TOP_*` | 同上 | 同上 |
| `LOCK_STATS_TOP_*` | 同上 | 同上 |

**実践的クエリ**:

```sql
-- CPU消費上位クエリ（直近1時間）
SELECT text, avg_cpu_seconds, avg_latency_seconds, execution_count,
       avg_rows_scanned, avg_rows_scanned / NULLIF(avg_rows_returned, 0) AS scan_ratio
FROM SPANNER_SYS.QUERY_STATS_TOP_HOUR
WHERE interval_end = (SELECT MAX(interval_end) FROM SPANNER_SYS.QUERY_STATS_TOP_HOUR)
ORDER BY avg_cpu_seconds DESC LIMIT 10;

-- ロック競合が多いキー範囲（直近1時間）
SELECT row_range_start_key, total_lock_wait_seconds, sample_lock_requests
FROM SPANNER_SYS.LOCK_STATS_TOP_HOUR
ORDER BY total_lock_wait_seconds DESC LIMIT 10;

-- アボート率が高いトランザクション
SELECT transaction_tag, commit_attempt_count, commit_abort_count,
       SAFE_DIVIDE(commit_abort_count, commit_attempt_count) AS abort_rate
FROM SPANNER_SYS.TXN_STATS_TOP_10MINUTE
ORDER BY abort_rate DESC LIMIT 10;
```

**トランザクションタグ・リクエストタグ**: 本番では必ず付与して `SPANNER_SYS` での絞り込みを容易にする。

### 4.4 アラート設計

**必須アラート**:

| アラート | 条件 | 理由 |
|---------|------|------|
| CPU（High Priority） | > 65% が5分継続 | フェイルオーバーマージン |
| Smoothed CPU | > 60% が24時間継続 | キャパシティプランニング |
| API レイテンシ P99 | SLO閾値の80%超え | バーンレート予兆 |
| エラーレート | UNAVAILABLE/DEADLINE_EXCEEDED > 1% | サービス影響 |
| ストレージ | 上限の70%超え | 容量計画 |

**アラートすべきでないもの**: 一時的CPUスパイク（1〜2分）、Low Priority CPU高騰単独、少量のNOT_FOUNDエラー。

### 4.5 よくある障害パターン

**DEADLINE_EXCEEDED**:
1. CPU過負荷 → `QUERY_STATS_TOP_MINUTE` で重量クエリ特定
2. 非効率クエリ → `avg_rows_scanned` >> `avg_rows_returned` のクエリを探す
3. タイムアウト設定 → **短すぎるタイムアウト + 即リトライはアンチパターン**

**ABORTED（トランザクション競合）**:
1. `TXN_STATS` でアボート率確認
2. `LOCK_STATS` でロック競合キー範囲特定
3. 対策: トランザクション保持時間最小化、読み取り専用トランザクション活用、楽観的ロック検討

**セッションプール枯渇**:
- `in_use_sessions` が `max_allowed_sessions` に張り付き → セッションリーク疑い
- Java: `setTrackStackTraceOfSessionCheckout(true)` で追跡

### 4.6 監査ログ

| ログ種別 | 対象 | デフォルト |
|---------|------|----------|
| Admin Activity | インスタンス/DB作成変更、IAM変更 | 常時有効（無効化不可） |
| Data Access | SQLクエリ、データ読み書き | **無効**（有効化が必要） |

Data Access Logsは大量に生成されコスト急増の可能性あり → 必要なサービスに限定して有効化。

## 5. クライアントライブラリパターン & 接続管理

### 5.1 セッション管理

| パラメータ | デフォルト | 推奨 |
|----------|----------|------|
| `MinSessions` | 100 | 予想される定常並列数 |
| `MaxSessions` | 400 | 最大同時トランザクション数 |
| `NumChannels` | 4 | `MaxSessions / 100` |

**Multiplexed Sessions**（2025年デフォルト有効）: 1セッションで大量の並列リクエスト処理。keep-alive不要、セッション管理簡略化。

```bash
export GOOGLE_CLOUD_SPANNER_MULTIPLEXED_SESSIONS=true
```

**セッションリーク検出**: `ListSessions` APIで最終使用時刻確認、OpenTelemetryでセッションプールメトリクス収集。

### 5.2 gRPC チャネルプール

- 1チャネルは **最大100並列リクエスト** を処理
- **Node.js は複数gRPCチャネル非対応** → 100セッション超なら複数クライアント作成

### 5.3 リトライ & エラーハンドリング

| エラーコード | リトライ | 説明 |
|------------|---------|------|
| `ABORTED` | **必須**（同一セッション内） | トランザクション競合 |
| `UNAVAILABLE` | 自動対象 | 一時的ネットワーク問題 |
| `DEADLINE_EXCEEDED` | 状況による | タイムアウト超過 |
| `NOT_FOUND` | 不可 | リソース不存在 |

**ABORTED はライブラリの `RunInTransaction` 内で自動リトライ。冪等性を確保すること。**

### 5.4 言語別の注意点

**Go**: `google-cloud-go/spanner` v1.86.x以降でMultiplexed Sessionsデフォルト有効。

**Java**: `setWriteSessionsFraction()` は非推奨（効果なし）。Spring Bootではシングルトンクライアント必須。

**Python**: デフォルト `BurstyPool`（高負荷時にセッション爆発の可能性）→ `FixedSizePool` 推奨。

**Node.js**: 複数gRPCチャネル非対応。`database.getTransaction()` 手動使用時はリリース忘れに注意。

### 5.5 リクエストプライオリティ

| プライオリティ | 用途 |
|--------------|------|
| `PRIORITY_HIGH` | OLTP、ユーザー向けリアルタイム処理 |
| `PRIORITY_MEDIUM` | デフォルト |
| `PRIORITY_LOW` | バッチ処理、分析クエリ |

```go
// 分析クエリはLOWプライオリティ
iter := client.Single().QueryWithOptions(ctx, stmt,
    spanner.QueryOptions{Priority: sppb.RequestOptions_PRIORITY_LOW})
```

### 5.6 GKE からの接続

- **Workload Identity** を使用（サービスアカウントキー禁止）
- GKEメタデータサーバーはノードあたり **最大500並列接続**
- 各Podの `MaxSessions` × レプリカ数がセッション上限（10,000/DB）を超えないよう設計

### 5.7 エミュレータ

```bash
docker run -p 9010:9010 -p 9020:9020 gcr.io/cloud-spanner-emulator/emulator
export SPANNER_EMULATOR_HOST="localhost:9010"
```

**制限**: 並列read-writeトランザクション不可（DB全体ロック）、メモリのみ（永続化なし）、SPANNER_SYS一部非対応、パフォーマンステスト不可。

## 6. コスト最適化・セキュリティ & 移行

### 6.1 コスト構造

| 要素 | 単価（リージョナル） |
|------|-------------------|
| 1ノード（1,000 PU） | 約 $0.90/時間 |
| 100 PU | 約 $0.09/時間 |
| ストレージ | 約 $0.30/GB/月 |
| マルチリージョン | リージョナルの約3〜8倍 |

**エディション（2024年9月〜）**: Standard（リージョナルのみ）、Enterprise（Graph/Vector/オートスケーラー）、Enterprise Plus（マルチリージョン/99.999% SLA）。

### 6.2 コスト最適化戦略

| 施策 | 効果 |
|------|------|
| CUD 3年 | コンピュートコスト **最大40%削減** |
| CUD 1年 | コンピュートコスト **20%削減** |
| オートスケーラー | 閑散時の余剰コスト削減 |
| Stale Reads | 読み取り負荷軽減 |
| インターリーブ | 分散JOIN回避 |
| TTL | ストレージコスト削減 |
| 100 PU インスタンス | 開発環境を月額 ~$65 に |

**CUD はベースライン（常時必要な最小PU）のみに購入。オートスケーラーのmax設定を忘れずに。**

### 6.3 セキュリティ

**CMEK（顧客管理暗号鍵）**:
- DB **作成時** に指定（後から変更不可）
- KMSキーはSpannerと **同じロケーション** に配置
- キーを誤って無効化/削除すると **データアクセス完全喪失**

**VPC Service Controls**:
- 必ず **DryRunモード** から開始
- Spanner単体でなく連携サービス（BigQuery, GCS, Dataflow）も同一境界内に

**Fine-Grained Access Control（FGAC）**:

```sql
CREATE ROLE inventory_reader;
GRANT SELECT ON TABLE Products TO ROLE inventory_reader;
GRANT SELECT (customer_name, order_id) ON TABLE Orders TO ROLE inventory_reader;
-- 機密カラム（credit_card_number）は除外
```

### 6.4 移行戦略

**Spanner Migration Tool（SMT）**: MySQL, PostgreSQL, SQL Server, Oracle, DynamoDBから移行支援。

**推奨移行フロー（最小ダウンタイム）**:

```
Phase 1: スキーマ変換・初期ロード
Phase 2: CDCで増分同期開始
Phase 3: デュアルライトモード（書き込み: ソース+Spanner、読み取り: ソース）
Phase 4: DVT（Data Validation Tool）で整合性検証
Phase 5: カットオーバー（Spanner完全移行）
Phase 6: リバースレプリケーション維持（フォールバック用）
```

**最大の落とし穴**: 連番主キー（AUTO_INCREMENT）→ UUID v4 / ビット反転シーケンスへの変換が必須。

### 6.5 PostgreSQL インターフェース

- DB作成時にダイアレクト選択（**後から変更不可**）
- DDLトランザクション非サポート
- 多くのPostgreSQL組み込み関数が未サポート
- **GoogleSQLが推奨**（移行の便宜以外では優位性なし）

### 6.6 統合パターン

| 統合先 | パターン | ユースケース |
|--------|---------|-------------|
| BigQuery | Change Streams + Dataflow | リアルタイム分析 |
| BigQuery | フェデレーテッドクエリ | 少量のリアルタイムルックアップ |
| Pub/Sub | Change Streams + Dataflow | イベント駆動アーキテクチャ |
| GCS | Change Streams + Dataflow | コンプライアンス監査ログ |
| Memorystore Redis | Cache-Aside | 高頻度読み取りのレイテンシ改善 |

| Change Stream 保持期間 | 最大 7日 |

| 統計情報の自動更新間隔 | 3日 |

### 障害発生時の初動チェックリスト

```
1. CPU 使用率と api/request_latencies の急変を確認
2. QUERY_STATS_TOP_MINUTE で直前の重量クエリを特定
3. TXN_STATS でアボート率が急上昇したトランザクションを確認
4. LOCK_STATS でロック競合キー範囲を確認
5. Key Visualizer でホットスポット発生の有無を確認
6. クライアント側メトリクス（OpenTelemetry）とサーバー側の差分を分析
```

### スキーマ設計チェックリスト

```
プライマリキー
  [ ] 先頭キーに単調増加値を使っていないか
  [ ] UUID v4 または ビット反転シーケンスを採用しているか
  [ ] UUIDv7 を使う場合、先頭に分散キーを配置しているか

インターリーブテーブル
  [ ] 主要アクセスが親子を一緒に操作するパターンか
  [ ] 親+子のデータ量が 8GB 以内か
  [ ] TTL を使う場合、全子テーブルが ON DELETE CASCADE か

DDL
  [ ] テーブルと初期インデックスを同一バッチで作成しているか
  [ ] バックフィル必要インデックスは1日3本以内か
  [ ] インデックス作成前にノードを増設しているか

カラム型
  [ ] 金融データに FLOAT64 を使っていないか（NUMERIC 必須）
  [ ] JSON 列の検索に生成カラム経由インデックスを計画しているか

セカンダリインデックス
  [ ] コミットタイムスタンプをインデックス先頭に置いていないか
  [ ] スパースカラムに NULL_FILTERED インデックスを使っているか

TTL
  [ ] PITR を有効化したか（version_retention_period 7日推奨）
  [ ] トゥームストーン問題のクエリフィルタを設計しているか
  [ ] リストア後のTTL再設定手順があるか
```
